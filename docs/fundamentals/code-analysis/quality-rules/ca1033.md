## <a name="cause"></a><span data-ttu-id="1b750-112">Przyczyna</span><span class="sxs-lookup"><span data-stu-id="1b750-112">Cause</span></span>

<span data-ttu-id="1b750-113">Niezapieczętowany typ widoczny na zewnątrz zapewnia jawną implementację metody interfejsu publicznego i nie dostarcza alternatywnej metody widocznej z zewnątrz o tej samej nazwie.</span><span class="sxs-lookup"><span data-stu-id="1b750-113">An unsealed externally visible type provides an explicit method implementation of a public interface and does not provide an alternative externally visible method that has the same name.</span></span>

## <a name="rule-description"></a><span data-ttu-id="1b750-114">Opis reguły</span><span class="sxs-lookup"><span data-stu-id="1b750-114">Rule description</span></span>

<span data-ttu-id="1b750-115">Rozważmy typ podstawowy, który jawnie implementuje metodę interfejsu publicznego.</span><span class="sxs-lookup"><span data-stu-id="1b750-115">Consider a base type that explicitly implements a public interface method.</span></span> <span data-ttu-id="1b750-116">Typ, który pochodzi od typu podstawowego, może uzyskać dostęp do dziedziczonej metody interfejsu tylko za pośrednictwem odwołania do bieżącego wystąpienia ( `this` w języku C#), który jest rzutowany do interfejsu.</span><span class="sxs-lookup"><span data-stu-id="1b750-116">A type that derives from the base type can access the inherited interface method only through a reference to the current instance (`this` in C#) that is cast to the interface.</span></span> <span data-ttu-id="1b750-117">Jeśli typ pochodny ponownie implementuje metodę dziedziczonego interfejsu (jawnie), nie można już uzyskać dostępu do implementacji podstawowej.</span><span class="sxs-lookup"><span data-stu-id="1b750-117">If the derived type reimplements (explicitly) the inherited interface method, the base implementation can no longer be accessed.</span></span> <span data-ttu-id="1b750-118">Wywołanie przez odwołanie do bieżącego wystąpienia wywoła implementację pochodną; powoduje to rekursję i ostateczne przepełnienie stosu.</span><span class="sxs-lookup"><span data-stu-id="1b750-118">The call through the current instance reference will invoke the derived implementation; this causes recursion and an eventual stack overflow.</span></span>

<span data-ttu-id="1b750-119">Ta zasada nie zgłasza naruszenia dla jawnej implementacji, <xref:System.IDisposable.Dispose%2A?displayProperty=fullName> gdy podano zewnętrznie widoczną `Close()` lub `System.IDisposable.Dispose(Boolean)` metodę.</span><span class="sxs-lookup"><span data-stu-id="1b750-119">This rule does not report a violation for an explicit implementation of <xref:System.IDisposable.Dispose%2A?displayProperty=fullName> when an externally visible `Close()` or `System.IDisposable.Dispose(Boolean)` method is provided.</span></span>

## <a name="how-to-fix-violations"></a><span data-ttu-id="1b750-120">Jak naprawić naruszenia</span><span class="sxs-lookup"><span data-stu-id="1b750-120">How to fix violations</span></span>

<span data-ttu-id="1b750-121">Aby naprawić naruszenie tej reguły, zaimplementuj nową metodę, która uwidacznia te same funkcje i jest widoczna dla typów pochodnych lub Zmień na niejawną implementację.</span><span class="sxs-lookup"><span data-stu-id="1b750-121">To fix a violation of this rule, implement a new method that exposes the same functionality and is visible to derived types or change to a nonexplicit implementation.</span></span> <span data-ttu-id="1b750-122">W przypadku akceptowalnej zmiany istotne jest, aby typ był zapieczętowany.</span><span class="sxs-lookup"><span data-stu-id="1b750-122">If a breaking change is acceptable, an alternative is to make the type sealed.</span></span>
