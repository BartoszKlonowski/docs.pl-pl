# <a name="apply-cqrs-and-cqs-approaches-in-a-ddd-microservice-in-eshoponcontainers"></a><span data-ttu-id="1795f-103">Stosowanie metod CQRS i CQS w mikrousługach DDD w eShopOnContainers</span><span class="sxs-lookup"><span data-stu-id="1795f-103">Apply CQRS and CQS approaches in a DDD microservice in eShopOnContainers</span></span>

<span data-ttu-id="1795f-104">Projekt mikrousługi zamawiania w aplikacji referencyjnej eShopOnContainers opiera się na zasadach CQRS.</span><span class="sxs-lookup"><span data-stu-id="1795f-104">The design of the ordering microservice at the eShopOnContainers reference application is based on CQRS principles.</span></span> <span data-ttu-id="1795f-105">Jednak używa najprostszego podejścia, które jest po prostu oddzielenie kwerend od poleceń i przy użyciu tej samej bazy danych dla obu akcji.</span><span class="sxs-lookup"><span data-stu-id="1795f-105">However, it uses the simplest approach, which is just separating the queries from the commands and using the same database for both actions.</span></span>

<span data-ttu-id="1795f-106">Istotą tych wzorców, a ważnym punktem tutaj jest to, że zapytania są idempotentne: bez względu na to, ile razy kwerendy systemu, stan tego systemu nie ulegnie zmianie.</span><span class="sxs-lookup"><span data-stu-id="1795f-106">The essence of those patterns, and the important point here, is that queries are idempotent: no matter how many times you query a system, the state of that system won't change.</span></span> <span data-ttu-id="1795f-107">Innymi słowy, zapytania są bez efektów ubocznych.</span><span class="sxs-lookup"><span data-stu-id="1795f-107">In other words, queries are side-effect free.</span></span>

<span data-ttu-id="1795f-108">W związku z tym można użyć innego modelu danych "odczyty" niż logiki transakcyjnej "zapisuje" model domeny, nawet jeśli mikrousługi zamawiania są przy użyciu tej samej bazy danych.</span><span class="sxs-lookup"><span data-stu-id="1795f-108">Therefore, you could use a different "reads" data model than the transactional logic "writes" domain model, even though the ordering microservices are using the same database.</span></span> <span data-ttu-id="1795f-109">W związku z tym jest to uproszczone podejście CQRS.</span><span class="sxs-lookup"><span data-stu-id="1795f-109">Hence, this is a simplified CQRS approach.</span></span>

<span data-ttu-id="1795f-110">Z drugiej strony polecenia, które wyzwalają transakcje i aktualizacje danych, zmieniają stan w systemie.</span><span class="sxs-lookup"><span data-stu-id="1795f-110">On the other hand, commands, which trigger transactions and data updates, change state in the system.</span></span> <span data-ttu-id="1795f-111">Dzięki poleciom musisz zachować ostrożność w radzeniu sobie ze złożonością i ciągle zmieniającymi się regułami biznesowymi.</span><span class="sxs-lookup"><span data-stu-id="1795f-111">With commands, you need to be careful when dealing with complexity and ever-changing business rules.</span></span> <span data-ttu-id="1795f-112">W tym miejscu chcesz zastosować techniki DDD, aby mieć lepiej modelowany system.</span><span class="sxs-lookup"><span data-stu-id="1795f-112">This is where you want to apply DDD techniques to have a better modeled system.</span></span>

<span data-ttu-id="1795f-113">Wzory DDD przedstawione w tym przewodniku nie powinny być stosowane powszechnie.</span><span class="sxs-lookup"><span data-stu-id="1795f-113">The DDD patterns presented in this guide should not be applied universally.</span></span> <span data-ttu-id="1795f-114">Wprowadzają one ograniczenia w projekcie.</span><span class="sxs-lookup"><span data-stu-id="1795f-114">They introduce constraints on your design.</span></span> <span data-ttu-id="1795f-115">Ograniczenia te zapewniają korzyści, takie jak wyższa jakość w czasie, zwłaszcza w poleceniach i innych kodów, który modyfikuje stan systemu.</span><span class="sxs-lookup"><span data-stu-id="1795f-115">Those constraints provide benefits such as higher quality over time, especially in commands and other code that modifies system state.</span></span> <span data-ttu-id="1795f-116">Jednak te ograniczenia dodać złożoność z mniejszą liczbą korzyści dla odczytu i wykonywania zapytań danych.</span><span class="sxs-lookup"><span data-stu-id="1795f-116">However, those constraints add complexity with fewer benefits for reading and querying data.</span></span>

<span data-ttu-id="1795f-117">Jednym z takich wzorów jest wzór Agregacji, który analizujemy więcej w kolejnych sekcjach.</span><span class="sxs-lookup"><span data-stu-id="1795f-117">One such pattern is the Aggregate pattern, which we examine more in later sections.</span></span> <span data-ttu-id="1795f-118">Krótko mówiąc, we wzorcu Agreguj wiele obiektów domeny traktujesz jako pojedynczą jednostkę w wyniku ich relacji w domenie.</span><span class="sxs-lookup"><span data-stu-id="1795f-118">Briefly, in the Aggregate pattern, you treat many domain objects as a single unit as a result of their relationship in the domain.</span></span> <span data-ttu-id="1795f-119">Nie zawsze można uzyskać korzyści z tego wzorca w kwerendach; może zwiększyć złożoność logiki zapytań.</span><span class="sxs-lookup"><span data-stu-id="1795f-119">You might not always gain advantages from this pattern in queries; it can increase the complexity of query logic.</span></span> <span data-ttu-id="1795f-120">W przypadku kwerend tylko do odczytu nie można uzyskać korzyści wynikające z traktowania wielu obiektów jako pojedynczego agregacji.</span><span class="sxs-lookup"><span data-stu-id="1795f-120">For read-only queries, you do not get the advantages of treating multiple objects as a single Aggregate.</span></span> <span data-ttu-id="1795f-121">Otrzymujesz tylko złożoność.</span><span class="sxs-lookup"><span data-stu-id="1795f-121">You only get the complexity.</span></span>

<span data-ttu-id="1795f-122">Jak pokazano na rysunku 7-2 w poprzedniej sekcji, w tym przewodniku sugeruje przy użyciu wzorców DDD tylko w transakcyjnych/aktualizuje obszar mikrousługi (czyli wyzwalane przez polecenia).</span><span class="sxs-lookup"><span data-stu-id="1795f-122">As shown in Figure 7-2 in the previous section, this guide suggests using DDD patterns only in the transactional/updates area of your microservice (that is, as triggered by commands).</span></span> <span data-ttu-id="1795f-123">Zapytania można wykonać prostsze podejście i powinny być oddzielone od poleceń, zgodnie z podejściem CQRS.</span><span class="sxs-lookup"><span data-stu-id="1795f-123">Queries can follow a simpler approach and should be separated from commands, following a CQRS approach.</span></span>

<span data-ttu-id="1795f-124">Do implementacji "po stronie zapytań", można wybrać między wieloma podejściami, z pełnowymiarowego ORM, takich jak EF Core, projekcje AutoMapper, procedury przechowywane, widoki, zmaterializowane widoki lub mikro ORM.</span><span class="sxs-lookup"><span data-stu-id="1795f-124">For implementing the "queries side", you can choose between many approaches, from your full-blown ORM like EF Core, AutoMapper projections, stored procedures, views, materialized views or a micro ORM.</span></span>

<span data-ttu-id="1795f-125">W tym przewodniku i w eShopOnContainers (w szczególności mikrousługi zamawiania) zdecydowaliśmy się zaimplementować proste zapytania za pomocą micro ORM jak [Dapper](https://github.com/StackExchange/dapper-dot-net).</span><span class="sxs-lookup"><span data-stu-id="1795f-125">In this guide and in eShopOnContainers (specifically the ordering microservice) we chose to implement straight queries using a micro ORM like [Dapper](https://github.com/StackExchange/dapper-dot-net).</span></span> <span data-ttu-id="1795f-126">Dzięki temu można zaimplementować wszelkie zapytania oparte na instrukcjach SQL, aby uzyskać najlepszą wydajność, dzięki lekkiej ramach z bardzo małym obciążeniem.</span><span class="sxs-lookup"><span data-stu-id="1795f-126">This lets you implement any query based on SQL statements to get the best performance, thanks to a light framework with very little overhead.</span></span>

<span data-ttu-id="1795f-127">Należy zauważyć, że podczas korzystania z tej metody, wszelkie aktualizacje modelu, które wpływają na sposób jednostek są utrwalone do bazy danych SQL również potrzebują oddzielnych aktualizacji zapytań SQL używanych przez Dapper lub inne oddzielne (non-EF) podejścia do wykonywania zapytań.</span><span class="sxs-lookup"><span data-stu-id="1795f-127">Note that when you use this approach, any updates to your model that impact how entities are persisted to a SQL database also need separate updates to SQL queries used by Dapper or any other separate (non-EF) approaches to querying.</span></span>

## <a name="cqrs-and-ddd-patterns-are-not-top-level-architectures"></a><span data-ttu-id="1795f-128">Wzorce CQRS i DDD nie są architekturami najwyższego poziomu</span><span class="sxs-lookup"><span data-stu-id="1795f-128">CQRS and DDD patterns are not top-level architectures</span></span>

<span data-ttu-id="1795f-129">Ważne jest, aby zrozumieć, że CQRS i większość wzorców DDD (takich jak warstwy DDD lub model domeny z agregatami) nie są stylami architektury, ale tylko wzorcami architektury.</span><span class="sxs-lookup"><span data-stu-id="1795f-129">It's important to understand that CQRS and most DDD patterns (like DDD layers or a domain model with aggregates) are not architectural styles, but only architecture patterns.</span></span> <span data-ttu-id="1795f-130">Mikrousługi, SOA i architektury opartej na zdarzeniach (EDA) są przykładami stylów architektonicznych.</span><span class="sxs-lookup"><span data-stu-id="1795f-130">Microservices, SOA, and event-driven architecture (EDA) are examples of architectural styles.</span></span> <span data-ttu-id="1795f-131">Opisują system wielu składników, takich jak wiele mikrousług.</span><span class="sxs-lookup"><span data-stu-id="1795f-131">They describe a system of many components, such as many microservices.</span></span> <span data-ttu-id="1795f-132">Wzorce CQRS i DDD opisują coś wewnątrz jednego systemu lub składnika; w tym przypadku coś wewnątrz mikrousługi.</span><span class="sxs-lookup"><span data-stu-id="1795f-132">CQRS and DDD patterns describe something inside a single system or component; in this case, something inside a microservice.</span></span>

<span data-ttu-id="1795f-133">Różne ograniczone konteksty (BC) będą wykorzystywać różne wzorce.</span><span class="sxs-lookup"><span data-stu-id="1795f-133">Different Bounded Contexts (BCs) will employ different patterns.</span></span> <span data-ttu-id="1795f-134">Mają różne obowiązki, a to prowadzi do różnych rozwiązań.</span><span class="sxs-lookup"><span data-stu-id="1795f-134">They have different responsibilities, and that leads to different solutions.</span></span> <span data-ttu-id="1795f-135">Warto podkreślić, że wymuszanie tego samego wzorca wszędzie prowadzi do awarii.</span><span class="sxs-lookup"><span data-stu-id="1795f-135">It is worth emphasizing that forcing the same pattern everywhere leads to failure.</span></span> <span data-ttu-id="1795f-136">Nie używaj wzorców CQRS i DDD wszędzie.</span><span class="sxs-lookup"><span data-stu-id="1795f-136">Do not use CQRS and DDD patterns everywhere.</span></span> <span data-ttu-id="1795f-137">Wiele podsystemów, kontrolerów domeny lub mikrousług są prostsze i można łatwiej zaimplementować przy użyciu prostych usług CRUD lub przy użyciu innego podejścia.</span><span class="sxs-lookup"><span data-stu-id="1795f-137">Many subsystems, BCs, or microservices are simpler and can be implemented more easily using simple CRUD services or using another approach.</span></span>

<span data-ttu-id="1795f-138">Istnieje tylko jedna architektura aplikacji: architektura systemu lub aplikacji end-to-end, które projektujesz (na przykład architektura mikrousług).</span><span class="sxs-lookup"><span data-stu-id="1795f-138">There is only one application architecture: the architecture of the system or end-to-end application you are designing (for example, the microservices architecture).</span></span> <span data-ttu-id="1795f-139">Jednak projekt każdego ograniczonego kontekstu lub mikrousług w ramach tej aplikacji odzwierciedla własne kompromisy i wewnętrzne decyzje projektowe na poziomie wzorców architektury.</span><span class="sxs-lookup"><span data-stu-id="1795f-139">However, the design of each Bounded Context or microservice within that application reflects its own tradeoffs and internal design decisions at an architecture patterns level.</span></span> <span data-ttu-id="1795f-140">Nie próbuj stosować tych samych wzorców architektonicznych, takich jak CQRS lub DDD wszędzie.</span><span class="sxs-lookup"><span data-stu-id="1795f-140">Do not try to apply the same architectural patterns like CQRS or DDD everywhere.</span></span>
