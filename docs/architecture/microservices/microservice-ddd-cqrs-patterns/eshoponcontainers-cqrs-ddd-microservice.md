# <a name="apply-cqrs-and-cqs-approaches-in-a-ddd-microservice-in-eshoponcontainers"></a><span data-ttu-id="a1c53-103">Stosowanie metod CQRS i CQS w mikrousługach DDD w eShopOnContainers</span><span class="sxs-lookup"><span data-stu-id="a1c53-103">Apply CQRS and CQS approaches in a DDD microservice in eShopOnContainers</span></span>

<span data-ttu-id="a1c53-106">Istotą tych wzorców i ważnym punktem jest to, że zapytania są idempotentne: bez względu na to, ile razy zapytań systemu, stan tego systemu nie zmieni.</span><span class="sxs-lookup"><span data-stu-id="a1c53-106">The essence of those patterns, and the important point here, is that queries are idempotent: no matter how many times you query a system, the state of that system won't change.</span></span> <span data-ttu-id="a1c53-107">Innymi słowy zapytania są wolne od efektów ubocznych.</span><span class="sxs-lookup"><span data-stu-id="a1c53-107">In other words, queries are side-effect free.</span></span>

<span data-ttu-id="a1c53-108">W związku z tym można użyć innego modelu danych "odczyty" niż logiki transakcyjnej "zapisuje" model domeny, mimo że mikrousług porządkowania są przy użyciu tej samej bazy danych.</span><span class="sxs-lookup"><span data-stu-id="a1c53-108">Therefore, you could use a different “reads” data model than the transactional logic “writes” domain model, even though the ordering microservices are using the same database.</span></span> <span data-ttu-id="a1c53-109">W związku z tym jest to uproszczone podejście CQRS.</span><span class="sxs-lookup"><span data-stu-id="a1c53-109">Hence, this is a simplified CQRS approach.</span></span>

<span data-ttu-id="a1c53-110">Z drugiej strony polecenia, które wyzwalają transakcje i aktualizacje danych, zmieniają stan w systemie.</span><span class="sxs-lookup"><span data-stu-id="a1c53-110">On the other hand, commands, which trigger transactions and data updates, change state in the system.</span></span> <span data-ttu-id="a1c53-111">Dzięki poleceń, trzeba być ostrożnym w kontaktach ze złożonością i ciągle zmieniających się reguł biznesowych.</span><span class="sxs-lookup"><span data-stu-id="a1c53-111">With commands, you need to be careful when dealing with complexity and ever-changing business rules.</span></span> <span data-ttu-id="a1c53-112">To jest, gdy chcesz zastosować techniki DDD mieć lepszy system modelowany.</span><span class="sxs-lookup"><span data-stu-id="a1c53-112">This is where you want to apply DDD techniques to have a better modeled system.</span></span>

<span data-ttu-id="a1c53-113">Wzorce DDD przedstawione w tym przewodniku nie powinny być stosowane uniwersalnie.</span><span class="sxs-lookup"><span data-stu-id="a1c53-113">The DDD patterns presented in this guide should not be applied universally.</span></span> <span data-ttu-id="a1c53-114">Wprowadzają ograniczenia w projekcie.</span><span class="sxs-lookup"><span data-stu-id="a1c53-114">They introduce constraints on your design.</span></span> <span data-ttu-id="a1c53-115">Ograniczenia te zapewniają korzyści, takie jak wyższa jakość w czasie, szczególnie w poleceniach i innym kodzie modyfikuje stan systemu.</span><span class="sxs-lookup"><span data-stu-id="a1c53-115">Those constraints provide benefits such as higher quality over time, especially in commands and other code that modifies system state.</span></span> <span data-ttu-id="a1c53-116">Jednak te ograniczenia dodać złożoność z mniejszą ilością korzyści dla odczytu i wykonywania zapytań danych.</span><span class="sxs-lookup"><span data-stu-id="a1c53-116">However, those constraints add complexity with fewer benefits for reading and querying data.</span></span>

<span data-ttu-id="a1c53-117">Jednym z takich wzorców jest zagregowany wzorzec, który badamy bardziej w kolejnych sekcjach.</span><span class="sxs-lookup"><span data-stu-id="a1c53-117">One such pattern is the Aggregate pattern, which we examine more in later sections.</span></span> <span data-ttu-id="a1c53-118">Krótko mówiąc, we wzorcu agregacji wiele obiektów domeny traktujesz jako pojedynczą jednostkę w wyniku ich relacji w domenie.</span><span class="sxs-lookup"><span data-stu-id="a1c53-118">Briefly, in the Aggregate pattern, you treat many domain objects as a single unit as a result of their relationship in the domain.</span></span> <span data-ttu-id="a1c53-119">Nie zawsze może uzyskać korzyści z tego wzorca w kwerendach; może zwiększyć złożoność logiki zapytania.</span><span class="sxs-lookup"><span data-stu-id="a1c53-119">You might not always gain advantages from this pattern in queries; it can increase the complexity of query logic.</span></span> <span data-ttu-id="a1c53-120">W przypadku zapytań tylko do odczytu nie można uzyskać zalety traktowania wielu obiektów jako pojedynczej agregacji.</span><span class="sxs-lookup"><span data-stu-id="a1c53-120">For read-only queries, you do not get the advantages of treating multiple objects as a single Aggregate.</span></span> <span data-ttu-id="a1c53-121">Otrzymujesz tylko złożoność.</span><span class="sxs-lookup"><span data-stu-id="a1c53-121">You only get the complexity.</span></span>

<span data-ttu-id="a1c53-122">Jak pokazano na rysunku 7-2 w poprzedniej sekcji, ten przewodnik sugeruje przy użyciu wzorców DDD tylko w obszarze transakcyjnych/aktualizacji mikrousługi (to jest, jak wyzwalane przez polecenia).</span><span class="sxs-lookup"><span data-stu-id="a1c53-122">As shown in Figure 7-2 in the previous section, this guide suggests using DDD patterns only in the transactional/updates area of your microservice (that is, as triggered by commands).</span></span> <span data-ttu-id="a1c53-123">Kwerendy mogą być zgodne z prostszym podejściem i powinny być oddzielone od poleceń, zgodnie z podejściem CQRS.</span><span class="sxs-lookup"><span data-stu-id="a1c53-123">Queries can follow a simpler approach and should be separated from commands, following a CQRS approach.</span></span>

<span data-ttu-id="a1c53-124">W celu zaimplementowania "po stronie zapytań", można wybrać jedną z wielu metod, z pełnowartościowego ORM jak EF Core, Projekcje AutoMapper, procedury przechowywane, widoki, zmaterializowane widoki lub mikro ORM.</span><span class="sxs-lookup"><span data-stu-id="a1c53-124">For implementing the “queries side”, you can choose between many approaches, from your full-blown ORM like EF Core, AutoMapper projections, stored procedures, views, materialized views or a micro ORM.</span></span>

<span data-ttu-id="a1c53-125">W tym przewodniku i eShopOnContainers (w szczególności mikrousługi zamawiania) zdecydowaliśmy się zaimplementować proste zapytania przy użyciu mikro ORM jak [Dapper](https://github.com/StackExchange/dapper-dot-net).</span><span class="sxs-lookup"><span data-stu-id="a1c53-125">In this guide and in eShopOnContainers (specifically the ordering microservice) we chose to implement straight queries using a micro ORM like [Dapper](https://github.com/StackExchange/dapper-dot-net).</span></span> <span data-ttu-id="a1c53-126">Dzięki temu można zaimplementować dowolne zapytanie na podstawie instrukcji SQL, aby uzyskać najlepszą wydajność, dzięki lekkiej struktury z bardzo małym obciążeniem.</span><span class="sxs-lookup"><span data-stu-id="a1c53-126">This lets you implement any query based on SQL statements to get the best performance, thanks to a light framework with very little overhead.</span></span>

<span data-ttu-id="a1c53-127">Należy zauważyć, że podczas korzystania z tej metody, wszelkie aktualizacje do modelu, które mają wpływ na sposób jednostek są utrwalione do bazy danych SQL również potrzebują oddzielnych aktualizacji do zapytań SQL używanych przez Dapper lub innych oddzielnych (non-EF) podejścia do wykonywania zapytań.</span><span class="sxs-lookup"><span data-stu-id="a1c53-127">Note that when you use this approach, any updates to your model that impact how entities are persisted to a SQL database also need separate updates to SQL queries used by Dapper or any other separate (non-EF) approaches to querying.</span></span>

<span data-ttu-id="a1c53-138">Istnieje tylko jedna architektura aplikacji: architektura systemu lub aplikacji end-to-end, którą projektujesz (na przykład architektura mikrousług).</span><span class="sxs-lookup"><span data-stu-id="a1c53-138">There is only one application architecture: the architecture of the system or end-to-end application you are designing (for example, the microservices architecture).</span></span> <span data-ttu-id="a1c53-139">Jednak projekt każdego ograniczonego kontekstu lub mikrousługi w tej aplikacji odzwierciedla własne kompromisy i wewnętrzne decyzje projektowe na poziomie wzorców architektury.</span><span class="sxs-lookup"><span data-stu-id="a1c53-139">However, the design of each Bounded Context or microservice within that application reflects its own tradeoffs and internal design decisions at an architecture patterns level.</span></span> <span data-ttu-id="a1c53-140">Nie próbuj stosować tych samych wzorców architektonicznych, takich jak CQRS lub DDD wszędzie.</span><span class="sxs-lookup"><span data-stu-id="a1c53-140">Do not try to apply the same architectural patterns like CQRS or DDD everywhere.</span></span>
