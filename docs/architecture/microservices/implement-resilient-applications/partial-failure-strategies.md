# <a name="strategies-to-handle-partial-failure"></a><span data-ttu-id="b9929-103">Strategie obsługi awarii częściowej</span><span class="sxs-lookup"><span data-stu-id="b9929-103">Strategies to handle partial failure</span></span>

<span data-ttu-id="b9929-104">Poniżej przedstawiono strategie postępowania w przypadku awarii częściowych.</span><span class="sxs-lookup"><span data-stu-id="b9929-104">Strategies for dealing with partial failures include the following.</span></span>

<span data-ttu-id="b9929-105">**Używaj komunikacji asynchronicznej (na przykład komunikacja oparta na komunikatach) między mikrousługami wewnętrznymi**.</span><span class="sxs-lookup"><span data-stu-id="b9929-105">**Use asynchronous communication (for example, message-based communication) across internal microservices**.</span></span> <span data-ttu-id="b9929-106">Zdecydowanie zaleca się, aby nie tworzyć długich łańcuchów synchronicznych wywołań HTTP w ramach wewnętrznych mikrousług, ponieważ niewłaściwy projekt będzie ostatecznie stanowić główną przyczynę nieprawidłowej awarii.</span><span class="sxs-lookup"><span data-stu-id="b9929-106">It's highly advisable not to create long chains of synchronous HTTP calls across the internal microservices because that incorrect design will eventually become the main cause of bad outages.</span></span> <span data-ttu-id="b9929-107">W przeciwieństwie do komunikacji frontonu między aplikacjami klienckimi a pierwszym poziomem mikrousług lub z szczegółowymi bramami interfejsu API zaleca się używanie tylko komunikacji asynchronicznej (opartej na komunikatach) po zakończeniu pierwszego cyklu żądania/odpowiedzi dla mikrousług wewnętrznych.</span><span class="sxs-lookup"><span data-stu-id="b9929-107">On the contrary, except for the front-end communications between the client applications and the first level of microservices or fine-grained API Gateways, it's recommended to use only asynchronous (message-based) communication once past the initial request/response cycle, across the internal microservices.</span></span> <span data-ttu-id="b9929-108">Spójność ostateczna i architektury sterowane zdarzeniami pomogą zminimalizować skutki działania programu Ripple.</span><span class="sxs-lookup"><span data-stu-id="b9929-108">Eventual consistency and event-driven architectures will help to minimize ripple effects.</span></span> <span data-ttu-id="b9929-109">Te podejścia powodują wymuszenie wyższego poziomu autonomii mikrousługi i w związku z tym uniemożliwiają zanotowanie tego problemu.</span><span class="sxs-lookup"><span data-stu-id="b9929-109">These approaches enforce a higher level of microservice autonomy and therefore prevent against the problem noted here.</span></span>

<span data-ttu-id="b9929-110">**Użyj ponownych prób przy użyciu wykładniczej wycofywania**.</span><span class="sxs-lookup"><span data-stu-id="b9929-110">**Use retries with exponential backoff**.</span></span> <span data-ttu-id="b9929-111">Ta technika pozwala uniknąć krótkich i sporadycznych błędów przez wykonanie wywołania ponawianie próby określoną liczbę razy, na wypadek gdy usługa była niedostępna tylko przez krótki czas.</span><span class="sxs-lookup"><span data-stu-id="b9929-111">This technique helps to avoid short and intermittent failures by performing call retries a certain number of times, in case the service was not available only for a short time.</span></span> <span data-ttu-id="b9929-112">Może to być spowodowane sporadycznymi problemami z siecią lub przeniesieniem mikrousługi/kontenera do innego węzła w klastrze.</span><span class="sxs-lookup"><span data-stu-id="b9929-112">This might occur due to intermittent network issues or when a microservice/container is moved to a different node in a cluster.</span></span> <span data-ttu-id="b9929-113">Jeśli jednak te ponowne próby nie są prawidłowo zaprojektowane z wyłącznikami, można pogłębić efekty programu Ripple, ostatecznie nawet powodując [odmowę usługi (DOS)](https://en.wikipedia.org/wiki/Denial-of-service_attack).</span><span class="sxs-lookup"><span data-stu-id="b9929-113">However, if these retries are not designed properly with circuit breakers, it can aggravate the ripple effects, ultimately even causing a [Denial of Service (DoS)](https://en.wikipedia.org/wiki/Denial-of-service_attack).</span></span>

<span data-ttu-id="b9929-114">Obejść **limity czasu sieci**.</span><span class="sxs-lookup"><span data-stu-id="b9929-114">**Work around network timeouts**.</span></span> <span data-ttu-id="b9929-115">Ogólnie rzecz biorąc klienci powinni być zaprojektowani tak, aby nie blokowały się w nieskończoność, a zawsze używać limitów czasu podczas oczekiwania na odpowiedź.</span><span class="sxs-lookup"><span data-stu-id="b9929-115">In general, clients should be designed not to block indefinitely and to always use timeouts when waiting for a response.</span></span> <span data-ttu-id="b9929-116">Użycie limitów czasu gwarantuje, że zasoby nigdy nie są powiązane w nieskończoność.</span><span class="sxs-lookup"><span data-stu-id="b9929-116">Using timeouts ensures that resources are never tied up indefinitely.</span></span>

<span data-ttu-id="b9929-117">**Użyj wzorca wyłącznika**.</span><span class="sxs-lookup"><span data-stu-id="b9929-117">**Use the Circuit Breaker pattern**.</span></span> <span data-ttu-id="b9929-118">W tym podejściu proces klienta śledzi liczbę żądań zakończonych niepowodzeniem.</span><span class="sxs-lookup"><span data-stu-id="b9929-118">In this approach, the client process tracks the number of failed requests.</span></span> <span data-ttu-id="b9929-119">W przypadku przekroczenia skonfigurowanego limitu wartość "wyłącznik" spowoduje, że dalsze próby zakończą się niepowodzeniem.</span><span class="sxs-lookup"><span data-stu-id="b9929-119">If the error rate exceeds a configured limit, a “circuit breaker” trips so that further attempts fail immediately.</span></span> <span data-ttu-id="b9929-120">(Jeśli duża liczba żądań kończy się niepowodzeniem, sugeruje to, że usługa jest niedostępna i że żądania wysyłania nie są bezpunktowe). Po upływie limitu czasu klient powinien ponowić próbę, a jeśli nowe żądania zakończą się pomyślnie, Zamknij wyłącznika.</span><span class="sxs-lookup"><span data-stu-id="b9929-120">(If a large number of requests are failing, that suggests the service is unavailable and that sending requests is pointless.) After a timeout period, the client should try again and, if the new requests are successful, close the circuit breaker.</span></span>

<span data-ttu-id="b9929-121">**Podaj rezerwy**.</span><span class="sxs-lookup"><span data-stu-id="b9929-121">**Provide fallbacks**.</span></span> <span data-ttu-id="b9929-122">W tym podejściu proces klienta wykonuje logikę rezerwową, gdy żądanie nie powiedzie się, takie jak zwrócenie danych w pamięci podręcznej lub wartości domyślnej.</span><span class="sxs-lookup"><span data-stu-id="b9929-122">In this approach, the client process performs fallback logic when a request fails, such as returning cached data or a default value.</span></span> <span data-ttu-id="b9929-123">Jest to podejście odpowiednie dla zapytań i jest bardziej skomplikowane dla aktualizacji lub poleceń.</span><span class="sxs-lookup"><span data-stu-id="b9929-123">This is an approach suitable for queries, and is more complex for updates or commands.</span></span>

<span data-ttu-id="b9929-124">**Ogranicz liczbę żądań umieszczonych w kolejce**.</span><span class="sxs-lookup"><span data-stu-id="b9929-124">**Limit the number of queued requests**.</span></span> <span data-ttu-id="b9929-125">Klienci powinni również wprowadzić górną granicę liczby oczekujących żądań wysyłanych przez mikrousługę klienta do określonej usługi.</span><span class="sxs-lookup"><span data-stu-id="b9929-125">Clients should also impose an upper bound on the number of outstanding requests that a client microservice can send to a particular service.</span></span> <span data-ttu-id="b9929-126">Jeśli limit został osiągnięty, najprawdopodobniej nie wskazuje to dodatkowych żądań, a te próby powinny natychmiast zakończyć się niepowodzeniem.</span><span class="sxs-lookup"><span data-stu-id="b9929-126">If the limit has been reached, it's probably pointless to make additional requests, and those attempts should fail immediately.</span></span> <span data-ttu-id="b9929-127">W warunkach implementacji zasady [izolacji Polly grodzi](https://github.com/App-vNext/Polly/wiki/Bulkhead) mogą służyć do spełnienia tego wymagania.</span><span class="sxs-lookup"><span data-stu-id="b9929-127">In terms of implementation, the Polly [Bulkhead Isolation](https://github.com/App-vNext/Polly/wiki/Bulkhead) policy can be used to fulfill this requirement.</span></span> <span data-ttu-id="b9929-128">To podejście jest zasadniczo przetwarzanie równoległe ograniczeniem <xref:System.Threading.SemaphoreSlim> jako implementacji.</span><span class="sxs-lookup"><span data-stu-id="b9929-128">This approach is essentially a parallelization throttle with <xref:System.Threading.SemaphoreSlim> as the implementation.</span></span> <span data-ttu-id="b9929-129">Umożliwia także "queue" poza grodzią.</span><span class="sxs-lookup"><span data-stu-id="b9929-129">It also permits a "queue" outside the bulkhead.</span></span> <span data-ttu-id="b9929-130">Można aktywnie powiększać nadmierne obciążenie nawet przed wykonaniem (na przykład, ponieważ pojemność jest uznawana za zapełnienie).</span><span class="sxs-lookup"><span data-stu-id="b9929-130">You can proactively shed excess load even before execution (for example, because capacity is deemed full).</span></span> <span data-ttu-id="b9929-131">Powoduje to, że odpowiedź na niektóre scenariusze awarii jest szybsza niż wyłącznika, ponieważ wyłącznik czeka na błędy.</span><span class="sxs-lookup"><span data-stu-id="b9929-131">This makes its response to certain failure scenarios faster than a circuit breaker would be, since the circuit breaker waits for the failures.</span></span> <span data-ttu-id="b9929-132">Obiekt BulkheadPolicy w [Polly](http://www.thepollyproject.org/) uwidacznia, jak pełen grodzię i kolejkę są, i oferuje zdarzenia w przepełnieniu, tak aby można go było również używać do tworzenia zautomatyzowanego skalowania w poziomie.</span><span class="sxs-lookup"><span data-stu-id="b9929-132">The BulkheadPolicy object in [Polly](http://www.thepollyproject.org/) exposes how full the bulkhead and queue are, and offers events on overflow so can also be used to drive automated horizontal scaling.</span></span>
