## <a name="container-design-principles"></a><span data-ttu-id="f574c-113">Zasady projektowania kontenerów</span><span class="sxs-lookup"><span data-stu-id="f574c-113">Container design principles</span></span>

<span data-ttu-id="f574c-116">Podczas projektowania obrazu kontenera w pliku Dockerfile zostanie wyświetlona definicja [ENTRYPOINT.](https://docs.docker.com/engine/reference/builder/#entrypoint)</span><span class="sxs-lookup"><span data-stu-id="f574c-116">When you design a container image, you'll see an [ENTRYPOINT](https://docs.docker.com/engine/reference/builder/#entrypoint) definition in the Dockerfile.</span></span> <span data-ttu-id="f574c-117">Definiuje proces, którego okres istnienia kontroluje okres istnienia kontenera.</span><span class="sxs-lookup"><span data-stu-id="f574c-117">This defines the process whose lifetime controls the lifetime of the container.</span></span> <span data-ttu-id="f574c-118">Po zakończeniu procesu kończy się cykl życia kontenera.</span><span class="sxs-lookup"><span data-stu-id="f574c-118">When the process completes, the container lifecycle ends.</span></span> <span data-ttu-id="f574c-119">Kontenery mogą reprezentować długotrwałe procesy, takie jak serwery sieci web, ale mogą również reprezentować procesy krótkotrwałe, takie jak zadania wsadowe, które wcześniej mogły zostać zaimplementowane jako usługi Azure [WebJobs.](https://github.com/Azure/azure-webjobs-sdk/wiki)</span><span class="sxs-lookup"><span data-stu-id="f574c-119">Containers might represent long-running processes like web servers, but can also represent short-lived processes like batch jobs, which formerly might have been implemented as Azure [WebJobs](https://github.com/Azure/azure-webjobs-sdk/wiki).</span></span>

<span data-ttu-id="f574c-120">Jeśli proces nie powiedzie się, kontener kończy się, a koordynator przejmuje.</span><span class="sxs-lookup"><span data-stu-id="f574c-120">If the process fails, the container ends, and the orchestrator takes over.</span></span> <span data-ttu-id="f574c-121">Jeśli koordynator został skonfigurowany do przechowywania pięciu wystąpień uruchomionych i jeden kończy się niepowodzeniem, koordynator utworzy inne wystąpienie kontenera, aby zastąpić proces nie powiodło się.</span><span class="sxs-lookup"><span data-stu-id="f574c-121">If the orchestrator was configured to keep five instances running and one fails, the orchestrator will create another container instance to replace the failed process.</span></span> <span data-ttu-id="f574c-122">W zadaniu wsadowym proces jest uruchamiany z parametrami.</span><span class="sxs-lookup"><span data-stu-id="f574c-122">In a batch job, the process is started with parameters.</span></span> <span data-ttu-id="f574c-123">Po zakończeniu procesu praca zostanie zakończona.</span><span class="sxs-lookup"><span data-stu-id="f574c-123">When the process completes, the work is complete.</span></span> <span data-ttu-id="f574c-124">Niniejsze wskazówki są drążenia w dół na koordynatorów, później.</span><span class="sxs-lookup"><span data-stu-id="f574c-124">This guidance drills-down on orchestrators, later on.</span></span>
