# <a name="logical-architecture-versus-physical-architecture"></a><span data-ttu-id="ba761-103">Architektura logiczna a architektura fizyczna</span><span class="sxs-lookup"><span data-stu-id="ba761-103">Logical architecture versus physical architecture</span></span>

<span data-ttu-id="ba761-104">Jest to przydatne w tym momencie, aby zatrzymywać i omawiać rozróżnienie między architekturą logiczną i architekturą fizyczną oraz jak ma to zastosowanie do projektowania aplikacji opartych na mikrousługach.</span><span class="sxs-lookup"><span data-stu-id="ba761-104">It's useful at this point to stop and discuss the distinction between logical architecture and physical architecture, and how this applies to the design of microservice-based applications.</span></span>

<span data-ttu-id="ba761-105">Aby rozpocząć, tworzenie mikrousług nie wymaga zastosowania żadnej konkretnej technologii.</span><span class="sxs-lookup"><span data-stu-id="ba761-105">To begin, building microservices doesn't require the use of any specific technology.</span></span> <span data-ttu-id="ba761-106">Na przykład kontenery platformy Docker nie są wymagane do utworzenia architektury opartej na mikrousługach.</span><span class="sxs-lookup"><span data-stu-id="ba761-106">For instance, Docker containers aren't mandatory to create a microservice-based architecture.</span></span> <span data-ttu-id="ba761-107">Te mikrousługi mogą być również uruchamiane jako zwykłe procesy.</span><span class="sxs-lookup"><span data-stu-id="ba761-107">Those microservices could also be run as plain processes.</span></span> <span data-ttu-id="ba761-108">Mikrousługi są architekturą logiczną.</span><span class="sxs-lookup"><span data-stu-id="ba761-108">Microservices is a logical architecture.</span></span>

<span data-ttu-id="ba761-109">Ponadto nawet wtedy, gdy mikrousługa może być fizycznie zaimplementowana jako jedna usługa, proces lub kontener (dla uproszczenia, to podejście wykonywane w początkowej wersji [eShopOnContainers](https://aka.ms/MicroservicesArchitecture)), ta różnica między mikrousługą biznesową a usługą fizyczną lub kontenerem nie musi być konieczna we wszystkich przypadkach podczas tworzenia dużej i złożonej aplikacji składającej się z wielu dziesiątek lub nawet setek usług.</span><span class="sxs-lookup"><span data-stu-id="ba761-109">Moreover, even when a microservice could be physically implemented as a single service, process, or container (for simplicity's sake, that's the approach taken in the initial version of [eShopOnContainers](https://aka.ms/MicroservicesArchitecture)), this parity between business microservice and physical service or container isn't necessarily required in all cases when you build a large and complex application composed of many dozens or even hundreds of services.</span></span>

<span data-ttu-id="ba761-110">Jest to sytuacja, w której istnieje różnica między architekturą logiczną aplikacji a architekturą fizyczną.</span><span class="sxs-lookup"><span data-stu-id="ba761-110">This is where there's a difference between an application's logical architecture and physical architecture.</span></span> <span data-ttu-id="ba761-111">Architektura logiczna i granice logiczne systemu nie zawsze mapują jeden do jednego na architekturę fizyczną lub rozmieszczenia.</span><span class="sxs-lookup"><span data-stu-id="ba761-111">The logical architecture and logical boundaries of a system do not necessarily map one-to-one to the physical or deployment architecture.</span></span> <span data-ttu-id="ba761-112">Może się tak zdarzyć, ale często nie.</span><span class="sxs-lookup"><span data-stu-id="ba761-112">It can happen, but it often doesn't.</span></span>

<span data-ttu-id="ba761-113">Chociaż można zidentyfikować pewne mikrousługi biznesowe lub ograniczone konteksty, nie oznacza to, że najlepszym sposobem wdrożenia jest zawsze utworzenie pojedynczej usługi (takiej jak ASP.NET Web API) lub jednego kontenera Docker dla każdej mikrousługi biznesowej.</span><span class="sxs-lookup"><span data-stu-id="ba761-113">Although you might have identified certain business microservices or Bounded Contexts, it doesn't mean that the best way to implement them is always by creating a single service (such as an ASP.NET Web API) or single Docker container for each business microservice.</span></span> <span data-ttu-id="ba761-114">Zasada mówiąca, że każda mikrousługa biznesowa musi być implementowana przy użyciu jednej usługi lub kontenera jest zbyt sztywna.</span><span class="sxs-lookup"><span data-stu-id="ba761-114">Having a rule saying each business microservice has to be implemented using a single service or container is too rigid.</span></span>

<span data-ttu-id="ba761-115">W związku z tym, mikrousługa biznesowa lub ograniczone kontekst jest architekturą logiczną, która może być zbieżna (lub nie) z architekturą fizyczną.</span><span class="sxs-lookup"><span data-stu-id="ba761-115">Therefore, a business microservice or Bounded Context is a logical architecture that might coincide (or not) with physical architecture.</span></span> <span data-ttu-id="ba761-116">Ważną kwestią jest to, że mikrousługa biznesowa lub ograniczone kontekst muszą być autonomiczne, umożliwiając, aby kod i stan były niezależnie w wersji, wdrożone i skalowane.</span><span class="sxs-lookup"><span data-stu-id="ba761-116">The important point is that a business microservice or Bounded Context must be autonomous by allowing code and state to be independently versioned, deployed, and scaled.</span></span>

<span data-ttu-id="ba761-117">Jak pokazano na rysunku 4-8, usługa biznesowa katalogu może składać się z kilku usług lub procesów.</span><span class="sxs-lookup"><span data-stu-id="ba761-117">As Figure 4-8 shows, the catalog business microservice could be composed of several services or processes.</span></span> <span data-ttu-id="ba761-118">Mogą to być wiele usług internetowego interfejsu API ASP.NET lub dowolnego innego rodzaju usług korzystających z protokołu HTTP lub dowolnego innego protokołu.</span><span class="sxs-lookup"><span data-stu-id="ba761-118">These could be multiple ASP.NET Web API services or any other kind of services using HTTP or any other protocol.</span></span> <span data-ttu-id="ba761-119">Co ważniejsze, usługi mogą współużytkować te same dane, o ile te usługi są spójne w odniesieniu do tej samej domeny biznesowej.</span><span class="sxs-lookup"><span data-stu-id="ba761-119">More importantly, the services could share the same data, as long as these services are cohesive with respect to the same business domain.</span></span>

<span data-ttu-id="ba761-126">W skrócie logicznej architekturze mikrousług nie zawsze jest konieczna zbieżność z fizyczną architekturą wdrażania.</span><span class="sxs-lookup"><span data-stu-id="ba761-126">In short, the logical architecture of microservices doesn't always have to coincide with the physical deployment architecture.</span></span> <span data-ttu-id="ba761-127">W tym przewodniku, gdy wspominamy mikrousługi, oznacza to, że firma lub logiczna mikrousługa, która może mapować do jednej lub kilku (fizycznych) usług.</span><span class="sxs-lookup"><span data-stu-id="ba761-127">In this guide, whenever we mention a microservice, we mean a business or logical microservice that could map to one or more (physical) services.</span></span> <span data-ttu-id="ba761-128">W większości przypadków jest to jedna usługa, ale może być większa.</span><span class="sxs-lookup"><span data-stu-id="ba761-128">In most cases, this will be a single service, but it might be more.</span></span>
