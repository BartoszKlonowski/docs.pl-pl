# <a name="creating-evolving-and-versioning-microservice-apis-and-contracts"></a><span data-ttu-id="bcecc-103">Tworzenie, rozwijanie i przechowywanie wersji interfejsów API i kontaktów w mikrousługach</span><span class="sxs-lookup"><span data-stu-id="bcecc-103">Creating, evolving, and versioning microservice APIs and contracts</span></span>

<span data-ttu-id="bcecc-104">Interfejs API mikrousług jest umową między usługą a jej klientami.</span><span class="sxs-lookup"><span data-stu-id="bcecc-104">A microservice API is a contract between the service and its clients.</span></span> <span data-ttu-id="bcecc-105">Będziesz mógł rozwijać mikrousługi niezależnie tylko wtedy, gdy nie złamiesz jego umowy interfejsu API, dlatego kontrakt jest tak ważne.</span><span class="sxs-lookup"><span data-stu-id="bcecc-105">You'll be able to evolve a microservice independently only if you do not break its API contract, which is why the contract is so important.</span></span> <span data-ttu-id="bcecc-106">Jeśli zmienisz umowę, będzie to miało wpływ na aplikacje klienckie lub bramę interfejsu API.</span><span class="sxs-lookup"><span data-stu-id="bcecc-106">If you change the contract, it will impact your client applications or your API Gateway.</span></span>

<span data-ttu-id="bcecc-107">Charakter definicji interfejsu API zależy od protokołu, którego używasz.</span><span class="sxs-lookup"><span data-stu-id="bcecc-107">The nature of the API definition depends on which protocol you're using.</span></span> <span data-ttu-id="bcecc-108">Na przykład jeśli używasz wiadomości (takich jak [AMQP),](https://www.amqp.org/)interfejs API składa się z typów komunikatów.</span><span class="sxs-lookup"><span data-stu-id="bcecc-108">For instance, if you're using messaging (like [AMQP](https://www.amqp.org/)), the API consists of the message types.</span></span> <span data-ttu-id="bcecc-109">Jeśli używasz usług HTTP i RESTful, interfejs API składa się z adresów URL oraz formatów JSON żądania i odpowiedzi.</span><span class="sxs-lookup"><span data-stu-id="bcecc-109">If you're using HTTP and RESTful services, the API consists of the URLs and the request and response JSON formats.</span></span>

<span data-ttu-id="bcecc-110">Jednak nawet jeśli masz na uwadze początkowej umowy, interfejs API usługi będzie musiał zmienić w czasie.</span><span class="sxs-lookup"><span data-stu-id="bcecc-110">However, even if you're thoughtful about your initial contract, a service API will need to change over time.</span></span> <span data-ttu-id="bcecc-111">W takim przypadku — a zwłaszcza jeśli interfejs API jest publicznym interfejsem API używanym przez wiele aplikacji klienckich — zazwyczaj nie można wymusić na wszystkich klientach uaktualnienia do nowej umowy interfejsu API.</span><span class="sxs-lookup"><span data-stu-id="bcecc-111">When that happens—and especially if your API is a public API consumed by multiple client applications — you typically can't force all clients to upgrade to your new API contract.</span></span> <span data-ttu-id="bcecc-112">Zazwyczaj należy stopniowo wdrażać nowe wersje usługi w taki sposób, że zarówno stare, jak i nowe wersje umowy serwisowej są uruchomione jednocześnie.</span><span class="sxs-lookup"><span data-stu-id="bcecc-112">You usually need to incrementally deploy new versions of a service in a way that both old and new versions of a service contract are running simultaneously.</span></span> <span data-ttu-id="bcecc-113">Dlatego ważne jest, aby mieć strategię dla wersji usługi.</span><span class="sxs-lookup"><span data-stu-id="bcecc-113">Therefore, it's important to have a strategy for your service versioning.</span></span>

<span data-ttu-id="bcecc-114">Gdy zmiany interfejsu API są małe, na przykład w przypadku dodawania atrybutów lub parametrów do interfejsu API, klienci, którzy używają starszego interfejsu API, powinni przełączać się i pracować z nową wersją usługi.</span><span class="sxs-lookup"><span data-stu-id="bcecc-114">When the API changes are small, like if you add attributes or parameters to your API, clients that use an older API should switch and work with the new version of the service.</span></span> <span data-ttu-id="bcecc-115">Może być możliwe podanie wartości domyślnych dla wszystkich brakujących atrybutów, które są wymagane, a klienci mogą być w stanie zignorować wszelkie dodatkowe atrybuty odpowiedzi.</span><span class="sxs-lookup"><span data-stu-id="bcecc-115">You might be able to provide default values for any missing attributes that are required, and the clients might be able to ignore any extra response attributes.</span></span>

<span data-ttu-id="bcecc-116">Jednak czasami trzeba wprowadzić główne i niezgodne zmiany w interfejsie API usługi.</span><span class="sxs-lookup"><span data-stu-id="bcecc-116">However, sometimes you need to make major and incompatible changes to a service API.</span></span> <span data-ttu-id="bcecc-117">Ponieważ może nie być w stanie wymusić aplikacji klienckich lub usług do natychmiastowego uaktualnienia do nowej wersji, usługa musi obsługiwać starsze wersje interfejsu API przez pewien okres.</span><span class="sxs-lookup"><span data-stu-id="bcecc-117">Because you might not be able to force client applications or services to upgrade immediately to the new version, a service must support older versions of the API for some period.</span></span> <span data-ttu-id="bcecc-118">Jeśli używasz mechanizmu opartego na HTTP, takiego jak REST, jednym z podejść jest osadzanie numeru wersji interfejsu API w adresie URL lub w nagłówku HTTP.</span><span class="sxs-lookup"><span data-stu-id="bcecc-118">If you're using an HTTP-based mechanism such as REST, one approach is to embed the API version number in the URL or into an HTTP header.</span></span> <span data-ttu-id="bcecc-119">Następnie można zdecydować między implementacji obu wersji usługi jednocześnie w ramach tego samego wystąpienia usługi lub wdrażania różnych wystąpień, które każdy obsługuje wersję interfejsu API.</span><span class="sxs-lookup"><span data-stu-id="bcecc-119">Then you can decide between implementing both versions of the service simultaneously within the same service instance, or deploying different instances that each handle a version of the API.</span></span> <span data-ttu-id="bcecc-120">Dobrym podejściem do tego jest [wzorzec mediatora](https://en.wikipedia.org/wiki/Mediator_pattern) (na przykład [biblioteka MediatR),](https://github.com/jbogard/MediatR)aby oddzielić różne wersje implementacji do niezależnych programów obsługi.</span><span class="sxs-lookup"><span data-stu-id="bcecc-120">A good approach for this is the [Mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) (for example, [MediatR library](https://github.com/jbogard/MediatR)) to decouple the different implementation versions into independent handlers.</span></span>

<span data-ttu-id="bcecc-121">Na koniec, jeśli używasz architektury REST, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) jest najlepszym rozwiązaniem do wersji usług i umożliwienie evolvable interfejsów API.</span><span class="sxs-lookup"><span data-stu-id="bcecc-121">Finally, if you're using a REST architecture, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) is the best solution for versioning your services and allowing evolvable APIs.</span></span>
