## <a name="declaring-a-message"></a><span data-ttu-id="7cebb-106">Deklarowanie komunikatu</span><span class="sxs-lookup"><span data-stu-id="7cebb-106">Declaring a message</span></span>

<span data-ttu-id="7cebb-107">W programie WCF Klasa `Stock` dla aplikacji handlowego rynku giełdowego może być zdefiniowana w następujący sposób:</span><span class="sxs-lookup"><span data-stu-id="7cebb-107">In WCF, a `Stock` class for a stock market trading application might be defined like the following example:</span></span>

```csharp
namespace TraderSys
{
    [DataContract]
    public class Stock
    {
        [DataMember]
        public int Id { get; set;}
        [DataMember]
        public string Symbol { get; set;}
        [DataMember]
        public string DisplayName { get; set;}
        [DataMember]
        public int MarketId { get; set; }
    }
}
```

<span data-ttu-id="7cebb-108">Aby zaimplementować równoważną klasę w protobuf, należy ją zadeklarować w pliku `.proto`.</span><span class="sxs-lookup"><span data-stu-id="7cebb-108">To implement the equivalent class in Protobuf, it must be declared in the `.proto` file.</span></span> <span data-ttu-id="7cebb-109">Kompilator `protoc` będzie następnie generować klasę .NET jako część procesu kompilacji.</span><span class="sxs-lookup"><span data-stu-id="7cebb-109">The `protoc` compiler will then generate the .NET class as part of the build process.</span></span>

```protobuf
syntax "proto3";

option csharp_namespace = "TraderSys";

message Stock {

    int32 id = 1;
    string symbol = 2;
    string display_name = 3;
    int32 market_id = 4;

}  
```

<span data-ttu-id="7cebb-110">Pierwszy wiersz deklaruje używaną wersję składni.</span><span class="sxs-lookup"><span data-stu-id="7cebb-110">The first line declares the syntax version being used.</span></span> <span data-ttu-id="7cebb-111">Wersja 3 języka została wydana w 2016 i jest zalecaną wersją dla usług gRPC Services.</span><span class="sxs-lookup"><span data-stu-id="7cebb-111">Version 3 of the language was released in 2016 and is the recommended version for gRPC services.</span></span>

<span data-ttu-id="7cebb-112">Wiersz `option csharp_namespace` określa przestrzeń nazw, która ma być używana dla wygenerowanych C# typów.</span><span class="sxs-lookup"><span data-stu-id="7cebb-112">The `option csharp_namespace` line specifies the namespace to be used for the generated C# types.</span></span> <span data-ttu-id="7cebb-113">Ta opcja zostanie zignorowana w przypadku skompilowania pliku `.proto` dla innych języków.</span><span class="sxs-lookup"><span data-stu-id="7cebb-113">This option will be ignored when the `.proto` file is compiled for other languages.</span></span> <span data-ttu-id="7cebb-114">Często pliki protobuf zawierają opcje specyficzne dla języka dla kilku języków.</span><span class="sxs-lookup"><span data-stu-id="7cebb-114">It is common for Protobuf files to contain language-specific options for several languages.</span></span>

<span data-ttu-id="7cebb-115">Definicja komunikatu `Stock` określa cztery pola, z których każdy ma typ, nazwę i numer pola.</span><span class="sxs-lookup"><span data-stu-id="7cebb-115">The `Stock` message definition specifies four fields, each with a type, a name, and a field number.</span></span>

## <a name="field-numbers"></a><span data-ttu-id="7cebb-116">Numery pól</span><span class="sxs-lookup"><span data-stu-id="7cebb-116">Field numbers</span></span>

<span data-ttu-id="7cebb-117">Numery pól są ważną częścią protobuf.</span><span class="sxs-lookup"><span data-stu-id="7cebb-117">Field numbers are an important part of Protobuf.</span></span> <span data-ttu-id="7cebb-118">Są one używane do identyfikowania pól w szyfrowanych danych binarnych, co oznacza, że nie mogą ulec zmianie z wersji na wersję usługi.</span><span class="sxs-lookup"><span data-stu-id="7cebb-118">They're used to identify fields in the binary encoded data, which means they can't change from version to version of your service.</span></span> <span data-ttu-id="7cebb-119">Zalety to zgodność z poprzednimi i progresywnymi.</span><span class="sxs-lookup"><span data-stu-id="7cebb-119">The advantage is that backward and forward compatibility is possible.</span></span> <span data-ttu-id="7cebb-120">Klienci i usługi będą po prostu ignorować numery pól, o których się nie wie, o ile nie jest obsługiwana możliwość braku wartości.</span><span class="sxs-lookup"><span data-stu-id="7cebb-120">Clients and services will simply ignore field numbers they don't know about, as long as the possibility of missing values is handled.</span></span>

<span data-ttu-id="7cebb-121">W formacie binarnym numer pola jest połączony z identyfikatorem typu.</span><span class="sxs-lookup"><span data-stu-id="7cebb-121">In the binary format, the field number is combined with a type identifier.</span></span> <span data-ttu-id="7cebb-122">Numery pól od 1 do 15 można zakodować ich typem jako pojedynczy bajt; liczby z 16 do 2047 mają 2 bajty.</span><span class="sxs-lookup"><span data-stu-id="7cebb-122">Field numbers from 1 to 15 can be encoded with their type as a single byte; numbers from 16 to 2047 take 2 bytes.</span></span> <span data-ttu-id="7cebb-123">Jeśli potrzebujesz więcej niż 2047 pól w komunikacie z dowolnego powodu, możesz przejść do wyższego poziomu.</span><span class="sxs-lookup"><span data-stu-id="7cebb-123">You can go higher if you need more than 2047 fields on a message for any reason.</span></span> <span data-ttu-id="7cebb-124">Identyfikatory pojedynczych bajtów dla numerów pól od 1 do 15 oferują lepszą wydajność, dlatego należy używać ich w przypadku najbardziej podstawowych, często używanych pól.</span><span class="sxs-lookup"><span data-stu-id="7cebb-124">The single byte identifiers for field numbers 1 to 15 offer better performance, so you should use them for the most basic, frequently used fields.</span></span>

## <a name="the-generated-code"></a><span data-ttu-id="7cebb-130">Wygenerowany kod</span><span class="sxs-lookup"><span data-stu-id="7cebb-130">The generated code</span></span>

<span data-ttu-id="7cebb-131">Podczas kompilowania aplikacji protobuf tworzy klasy dla poszczególnych komunikatów, mapując typy natywne na C# typy.</span><span class="sxs-lookup"><span data-stu-id="7cebb-131">When you build your application, Protobuf creates classes for each of your messages, mapping its native types to C# types.</span></span> <span data-ttu-id="7cebb-132">Wygenerowany typ `Stock` powinien mieć następującą sygnaturę:</span><span class="sxs-lookup"><span data-stu-id="7cebb-132">The generated `Stock` type would have the following signature:</span></span>

```csharp

    public int Id { get; set; }

    public string DisplayName { get; set; }
    public int MarketId { get; set; }
}
```

<span data-ttu-id="7cebb-133">Wygenerowany kod jest znacznie bardziej skomplikowany niż ten, ponieważ każda klasa zawiera cały kod niezbędny do serializacji i deserializacji siebie w formacie sieci binarnej.</span><span class="sxs-lookup"><span data-stu-id="7cebb-133">The actual code that is generated is far more complicated than this, because each class contains all the code necessary to serialize and deserialize itself to the binary wire format.</span></span>
