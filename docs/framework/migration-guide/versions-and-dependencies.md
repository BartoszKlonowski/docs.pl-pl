# <a name="net-framework-versions-and-dependencies"></a>Wersje i zależności programu .NET Framework
Każda wersja programu.NET Framework zawiera środowisko uruchomieniowe języka wspólnego (CLR), biblioteki klas bazowych i inne zarządzane biblioteki. W tym temacie opisano kluczowe funkcje programu .NET Framework według wersji, podano informacje o bazowych wersjach środowiska CLR i skojarzonych środowiskach deweloperskich oraz zidentyfikowano wersje, które są instalowane przez system operacyjny Windows.  
  
> [!NOTE]
>  Aby uzyskać informacje dotyczące pobierania i instalacji programu .NET Framework, zobacz [Zainstaluj program .NET Framework dla deweloperów](../../../docs/framework/install/guide-for-developers.md).  
  
 Poniższa tabela zawiera podsumowanie Historia wersji .NET Framework i są powiązane z każdej wersji programu Visual Studio, Windows i Windows Server. Należy zauważyć, że program Visual Studio obsługuje określanie wielu platform docelowych kodu, więc nie trzeba ograniczać się do wersji programu .NET Framework, która jest aktualnie wyświetlona.  
  
 Każda nowa wersja programu .NET Framework zachowuje funkcje z poprzednich wersji i dodaje nowe funkcje. Środowisko CLR jest identyfikowane za pomocą własnego numeru wersji. Numer wersji programu .NET Framework jest zwiększany przy każdym wydaniu, chociaż wersja środowiska CLR nie zawsze jest zwiększana. Na przykład, .NET Framework 4 4.5 lub nowszej wersji 4 CLR, ale programu .NET Framework 2.0, 3.0 i 3.5 obejmują CLR 2.0. (Nie było wersji 3 środowiska CLR).  
  
 Zobacz [wymagania systemowe](../../../docs/framework/get-started/system-requirements.md) pełną listę obsługiwanych systemów operacyjnych. Pliki do pobrania, zobacz [Zainstaluj program .NET Framework dla deweloperów](../../../docs/framework/install/guide-for-developers.md). W celu określenia, które wersje programu .NET Framework są zainstalowane na komputerze, zobacz [porady: ustalić, które .NET Framework są zainstalowane wersje](../../../docs/framework/migration-guide/how-to-determine-which-versions-are-installed.md).

## <a name="targeting-and-running-net-framework-apps-for-version-45-and-later"></a>Aplikacje elementów docelowych i systemem .NET Framework w wersji 4.5 lub nowszy  
 [!INCLUDE[net_v45](../../../includes/net-v45-md.md)] Jest dostępna aktualizacja w miejscu, które zastępuje [!INCLUDE[net_v40_short](../../../includes/net-v40-short-md.md)] na komputerze i podobnie [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] 4.5.2, 4.6, 4.6.1, 4.6.2, 4.7 i 4.7.1 są aktualizacje w miejscu [!INCLUDE[net_v45](../../../includes/net-v45-md.md)], co oznacza, że korzystają z tej samej wersji środowiska wykonawczego Jednak wersji zestawu są aktualizowane i zawiera nowe typy i składniki. Po zakończeniu instalacji tych aktualizacji z [!INCLUDE[net_v40_short](../../../includes/net-v40-short-md.md)], [!INCLUDE[net_v45](../../../includes/net-v45-md.md)], .NET Framework 4.6 lub aplikacji .NET Framework 4.7 nadal uruchamiać bez konieczności ponownej kompilacji. Jednak odwrotna zależność nie istnieje. Nie zaleca się uruchamianie aplikacji przeznaczonych nowszej wersji programu .NET Framework w starszej wersji programu .NET Framework. Na przykład, nie zaleca się uruchamiania aplikacji elementy docelowe [!INCLUDE[net_v46](../../../includes/net-v46-md.md)] na [!INCLUDE[net_v45](../../../includes/net-v45-md.md)]. Należy przestrzegać następujących wytycznych:  
  
-   W programie Visual Studio, można wybrać [!INCLUDE[net_v45](../../../includes/net-v45-md.md)] jako platforma docelowa projektu (to ustawienie <xref:Microsoft.Build.Tasks.GetReferenceAssemblyPaths.TargetFrameworkMoniker%2A?displayProperty=nameWithType> właściwości) do skompilowania projektu jako [!INCLUDE[net_v45](../../../includes/net-v45-md.md)] zestawu lub pliku wykonywalnego. Tego zestawu lub pliku wykonywalnego następnie można używać na dowolnym komputerze, który ma [!INCLUDE[net_v45](../../../includes/net-v45-md.md)], 4.5.1, 4.5.2, 4.6, 4.6.1, 4.6.2, 4.7 lub 4.7.1 zainstalowane.  
  
-   W programie Visual Studio, można wybrać [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] jako platforma docelowa projektu (to ustawienie <xref:Microsoft.Build.Tasks.GetReferenceAssemblyPaths.TargetFrameworkMoniker%2A?displayProperty=nameWithType> właściwości) do skompilowania projektu jako [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] zestawu lub pliku wykonywalnego. Tego zestawu lub pliku wykonywalnego powinny być uruchamiane tylko na komputerach, które mają [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] lub nowsza wersja programu .NET Framework zainstalowana. Plik wykonywalny, którego celem jest [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] zostanie zablokowane uruchomiony na komputerze, który zawiera tylko starszej wersji programu .NET Framework, takich jak [!INCLUDE[net_v45](../../../includes/net-v45-md.md)], zainstalowane, i użytkownik zostanie monit o zainstalowanie [!INCLUDE[net_v451](../../../includes/net-v451-md.md)]. Ponadto [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] zestawów nie powinna być wywoływana z aplikacji, która jest przeznaczony dla starszej wersji programu .NET Framework, takich jak [!INCLUDE[net_v45](../../../includes/net-v45-md.md)].  
  
     [!INCLUDE[net_v451](../../../includes/net-v451-md.md)] i [!INCLUDE[net_v45](../../../includes/net-v45-md.md)] są używane tylko jako przykłady. Ta zasada ma zastosowanie do dowolnej aplikacji którego element docelowy nowszej wersji programu .NET Framework niż zainstalowana na komputerze, na którym jest uruchomiony.  
  
 Niektóre zmiany w programie .NET Framework mogą wymagać zmian w kodzie aplikacji; zobacz [zgodności aplikacji](../../../docs/framework/migration-guide/application-compatibility.md) przed uruchomieniem istniejące aplikacje za pomocą [!INCLUDE[net_v45](../../../includes/net-v45-md.md)] lub nowszy. Aby uzyskać więcej informacji na temat instalowania bieżącej wersji, zobacz [Zainstaluj program .NET Framework dla deweloperów](../../../docs/framework/install/guide-for-developers.md). Aby uzyskać informacje o pomocy technicznej dla programu .NET Framework, zobacz [obsługuje zasady cyklu pomocy firmy Microsoft .NET Framework](http://go.microsoft.com/fwlink/?LinkId=196607) w witrynie sieci Web Microsoft Support.  
  
## <a name="targeting-and-running-apps-for-older-versions"></a>Określanie platformy docelowej i uruchamianie aplikacji dla starszych wersji  
 Wersje 2.0, 3.0 i 3.5 programu .NET Framework zostały utworzone z użyciem tej samej wersji środowiska CLR (CLR 2.0). Te wersje reprezentują kolejne warstwy jednej instalacji. Każda wersja jest kompilowana przyrostowo na poprzednich wersjach. Nie jest możliwe uruchomienie wersji 2.0, 3.0 i 3.5 równocześnie na komputerze. Po zainstalowaniu wersji 3.5 automatycznie są udostępniane warstwy 2.0 i 3.0, a aplikacje, które zostały zaprojektowane dla wersji 2.0, 3.0 i 3.5, można uruchamiać z użyciem wersji 3.5. Jednak programu .NET Framework 4 kończy się tą metodą Tworzenie warstw. Począwszy od programu .NET Framework 4, można użyć wewnątrzprocesowego hostingu równoczesnego w celu uruchamiania wielu wersji środowiska CLR w pojedynczym procesie. Aby uzyskać więcej informacji, zobacz [zestawy i wykonywanie Side-by-Side](../../../docs/framework/app-domains/assemblies-and-side-by-side-execution.md).  
  
 Ponadto, jeśli aplikacja jest przeznaczony dla wersji 2.0, 3.0 lub 3.5, użytkowników może być konieczna, Włącz program .NET Framework 3.5 [!INCLUDE[win8](../../../includes/win8-md.md)] lub [!INCLUDE[win81](../../../includes/win81-md.md)] komputera przed uruchomieniem aplikacji. Aby uzyskać więcej informacji, zobacz [zainstalować program .NET Framework 3.5 w systemie Windows 10, Windows 8.1 i Windows 8](../../../docs/framework/install/dotnet-35-windows-10.md).
