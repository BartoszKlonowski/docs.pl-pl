# <a name="entity-data-model"></a><span data-ttu-id="cc432-102">Entity Data Model</span><span class="sxs-lookup"><span data-stu-id="cc432-102">Entity Data Model</span></span>
<span data-ttu-id="cc432-103">Entity Data Model (EDM) to zbiór pojęcia, które opisują struktury danych, niezależnie od tego, w postaci przechowywane.</span><span class="sxs-lookup"><span data-stu-id="cc432-103">The Entity Data Model (EDM) is a set of concepts that describe the structure of data, regardless of its stored form.</span></span> <span data-ttu-id="cc432-104">EDM obiektowy Model Relacja jednostki, opisanego przez Peter Chen w 1976, ale również jest oparta na modelu jednostki relacji i rozszerza jego tradycyjnych zastosowań.</span><span class="sxs-lookup"><span data-stu-id="cc432-104">The EDM borrows from the Entity-Relationship Model described by Peter Chen in 1976, but it also builds on the Entity-Relationship Model and extends its traditional uses.</span></span>  
  
 <span data-ttu-id="cc432-105">EDM rozwiązuje problemy, które wynikają z konieczności — dane przechowywane w wielu formularzach.</span><span class="sxs-lookup"><span data-stu-id="cc432-105">The EDM addresses the challenges that arise from having data stored in many forms.</span></span> <span data-ttu-id="cc432-106">Rozważmy na przykład firma, która przechowuje dane w relacyjnych baz danych, pliki tekstowe, pliki XML, arkusze kalkulacyjne i raporty.</span><span class="sxs-lookup"><span data-stu-id="cc432-106">For example, consider a business that stores data in relational databases, text files, XML files, spreadsheets, and reports.</span></span> <span data-ttu-id="cc432-107">Przedstawia informacje o istotnym wyzwaniom związanym z modelowania danych, projektowania aplikacji i dostępu data-ttu-id="cc432-108">Podczas projektowania aplikacji zorientowanych na dane, wezwanie jest do pisania kodu, wydajnego i łatwego w utrzymaniu bez obniżania oczekiwanego poziomu dostępu do danych wydajne, magazynu i skalowalności.</span><span class="sxs-lookup"><span data-stu-id="cc432-108">When designing a data-oriented application, the challenge is to write efficient and maintainable code without sacrificing efficient data access, storage, and scalability.</span></span> <span data-ttu-id="cc432-109">Przypadku relacyjnej struktury danych, dostęp do danych, magazynu i skalowalności jest access, storage, and scalability are very efficient, but writing efficient and maintainable code becomes more difficult.</span></span> <span data-ttu-id="cc432-110">Gdy dane znajdują się struktury obiektu, zostały cofnięte wad i zalet: Pisanie kodu wydajnego i łatwego w utrzymaniu pochodzi kosztem dostępu do danych wydajne, magazynu i skalowalności.</span><span class="sxs-lookup"><span data-stu-id="cc432-110">When data has an object structure, the trade-offs are reversed: Writing efficient and maintainable code comes at the cost of efficient data access, storage, and scalability.</span></span> <span data-ttu-id="cc432-111">Nawet wtedy, gdy można znaleźć równowagę między te wad, nowe wyzwania wystąpić, gdy dane są przenoszone z jednego formularza do innego.</span><span class="sxs-lookup"><span data-stu-id="cc432-111">Even if the right balance between these trade-offs can be found, new challenges arise when data is moved from one form to another.</span></span> <span data-ttu-id="cc432-112">W modelu Entity Data Model uwzględniają te problemy, zawierająca opis struktury danych pod względem jednostek i relacji, które są niezależne od dowolnego schematu magazynu.</span><span class="sxs-lookup"><span data-stu-id="cc432-112">The Entity Data Model addresses these challenges by describing the structure of data in terms of entities and relationships that are independent of any storage schema.</span></span> <span data-ttu-id="cc432-113">Dzięki temu formularzu przechowywanych danych nieodpowiednie do projektowania aplikacji i tworzenia.</span><span class="sxs-lookup"><span data-stu-id="cc432-113">This makes the stored form of data irrelevant to application design and development.</span></span> <span data-ttu-id="cc432-114">Ponadto ponieważ jednostek i relacji opisać struktury danych, jest używany w aplikacji (nie jego przechowywanej formularz), ich ewolucji miarę rozwoju aplikacji.</span><span class="sxs-lookup"><span data-stu-id="cc432-114">And, because entities and relationships describe the structure of data as it is used in an application (not its stored form), they can evolve as an application evolves.</span></span>  
  
 <span data-ttu-id="cc432-115">A `conceptual model` to reprezentacja określonej struktury danych jako jednostek i relacji i zwykle jest zdefiniowany w języka specyficznego dla domeny (DSL), który implementuje koncepcji EDM.</span><span class="sxs-lookup"><span data-stu-id="cc432-115">A `conceptual model` is a specific representation of the structure of data as entities and relationships, and data-stu-id="cc432-116">[Conceptual schema definition language (CSDL)](../../../../docs/framework/data/adonet/ef/language-reference/csdl-specification.md) is an example of such a domain-specific language.</span></span> <span data-ttu-id="cc432-117">Jednostek i relacji, które opisano w modelu koncepcyjnym mogą być uważane za abstrakcje obiektów i skojarzenia w aplikacji.</span><span class="sxs-lookup"><span data-stu-id="cc432-117">Entities and relationships described in a conceptual model can be thought of as abstractions of objects and associations in an application.</span></span> <span data-ttu-id="cc432-118">Umożliwia deweloperom skoncentrowanie się na modelu koncepcyjnego zapewnieniu schemat magazynu i pozwala na ich wydajność i łatwość konserwacji należy pamiętać, do pisania kodu.</span><span class="sxs-lookup"><span data-stu-id="cc432-118">This allows developers to focus on the conceptual model without concern for the storage schema, and allows them to write code with efficiency and maintainability in mind.</span></span> <span data-ttu-id="cc432-119">W międzyczasie projektantów schematu magazynu skupić się na wydajność dostępu do danych, magazynu i skalowalności.</span><span class="sxs-lookup"><span data-stu-id="cc432-119">Meanwhile storage schema designers can focus on the efficiency of data access, storage, and scalability.</span></span>
