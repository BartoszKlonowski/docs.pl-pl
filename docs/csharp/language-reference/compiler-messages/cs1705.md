# <a name="compiler-error-cs1705"></a>Błąd kompilatora CS1705
"TypeName", który ma wyższą wersję niż przywoływany zestaw "AssemblyName2" korzysta z zestawu "AssemblyName1"  
  
 Uzyskujesz dostęp do typu, który ma wyższy numer wersji niż numer wersji w zestawie odwołania. Zazwyczaj ten błąd jest spowodowany przypadkowym użytkowania dwie wersje tego samego zestawu.  
  
 Na przykład załóżmy, że masz dwa zestawy Asmb1 i Asmb2. Asmb1 zestawu odwołuje się do zestawu Asmb2 w wersji 1.0. Zestaw Asmb1 używa również klasę `MyClass` który został dodany do zestawu Asmb2 w wersji 2.0. Kompilator ma zasady ujednolicenie odwołań do powiązań i odwołania do `MyClass` w wersji 2.0 nie mogą być spełnione przez wersję 1.0.  
  
## <a name="example"></a>Przykład  
 Poniższy przykład bardziej szczegółowe składa się z czterech modułów kodu:  
  
-   Dwa pliki dll, które są identyczne, z wyjątkiem atrybutu wersji.  
  
-   Trzeci biblioteki DLL, który odwołuje się do dwóch pierwszych.  
  
-   Klient, który odwołuje się tylko wersję 1.0 identyczne bibliotek DLL, ale uzyskuje dostęp do klasy z wersji 2.0.  
  
 Poniższy kod tworzy pierwszy identyczne biblioteki dll. Aby uzyskać informacje o tym, jak można wygenerować plik klucza, zobacz [/KeyFile (opcje kompilatora C#)](../../../csharp/language-reference/compiler-options/keyfile-compiler-option.md).  
  
```csharp  
// CS1705a.cs  
  
// Compile by using the following command:   
//      csc /target:library /out:C:\\CS1705.dll /keyfile:mykey.snk CS1705a.cs  
  
// The DLL is created in the C:\ directory.  
  
// The AssemblyVersion attribute specifies version 1.0 for this DLL.  
  
[assembly:System.Reflection.AssemblyVersion("1.0")]  
public class Class1   
{  
   public void Method1() {}  
}  
```  
  
## <a name="example"></a>Przykład  
 Poniższy kod definiuje wersja 2.0 zestawu, określony przez <xref:System.Reflection.AssemblyVersionAttribute> atrybutu.  
  
```csharp  
// CS1705b.cs  
  
// Compile by using the following command:   
//     csc /target:library /out:CS1705.dll /keyfile:mykey.snk CS1705b.cs  
  
// The DLL is created in the current directory.  
  
// The AssemblyVersion attribute specifies version 2.0 for this DLL.  
  
[assembly:System.Reflection.AssemblyVersion("2.0")]  
public class Class1  
{  
    public void Method1() { }  
}  
```  
  
## <a name="example"></a>Przykład  
 Poniższy kod odwołuje się dwie wersje pliku DLL, które są zdefiniowane w poprzednim kodzie. `AssemblyA` Odwołanie do biblioteki DLL utworzony przez CS1705a.cs (w wersji 1.0). `AssemblyB` Odwołanie do biblioteki DLL utworzony przez CS1705b.cs (w wersji 2.0). W `ClassC`, dwie metody zostały zdefiniowane. Pierwsza strona, `Return1A`, zwraca obiekt typu `Class1A`, która jest aliasem `Class1` w wersji 1.0 biblioteki DLL. Druga Strona, `Return1B`, zwraca obiekt typu `Class1B`, która jest aliasem `Class1` z wersji 2.0, biblioteki dll. Definicja `Return1A` tworzy zależność w wersji 1.0; definicja `Return1B` tworzy zależność w wersji 2.0.  
  
```csharp  
// CS1705c.cs  
  
// Compile by using the following command. AssemblyA refers to the DLL created by  
// CS1705a.cs (version 1.0). AssemblyB refers to the DLL created by CS1705b.cs  
// (version 2.0).  
//    csc /target:library /r:AssemblyA=C:\\CS1705.dll /r:AssemblyB=CS1705.dll CS1705c.cs  
  
extern alias AssemblyA;  
extern alias AssemblyB;  
  
// Class1A is an alias for type Class1 from VS1705a.cs, which is in version 1.0   
// of the assembly. Class1B is an alias for type Class1 from CS1705b.cs, which  
// is in version 2.0 of the assembly.  
  
using Class1A = AssemblyA::Class1;  
using Class1B = AssemblyB::Class1;  
  
// Method Return1A in ClassC returns an object of type Class1A, which is  
// Class1 from version 1.0 of the DLL. Method Return1B returns an object  
// of type Class1B, which is Class1 from version 2.0 of the DLL.  
  
public class ClassC  
{  
    // The following line creates a dependency on version 1.0 of CS1705.dll.  
    // This is not the source of the problem when ClassC is accessed from  
    // CS1705d.cs because CS1705d.cs references version 1.0 of the DLL.   
    // Therefore, type Class1A and the assembly have the same version.  
    public static Class1A Return1A() { return new Class1A(); }  
  
    // The following line creates a dependency on version 2.0 of CS1705.dll.  
    // This causes compiler error CS1705 when ClassC is accessed from     
    // CS1705d.cs, because CS1705d.cs does not reference version 2.0 of the   
    // DLL. Class1B is the alias for Class1 in version 2.0, and CS1705d.cs   
    // references version 1.0.  
    public static Class1B Return1B() { return new Class1B(); }  
}  
```  
  
## <a name="example"></a>Przykład  
 Poniższy kod generuje błąd kompilatora CS1705. Odwołuje się DLL utworzony przez CS1705a.cs (w wersji 1.0). Jednak w `Main` uzyskuje dostęp do kodu w metodzie `ClassC` z CS1705c.cs. `ClassC` korzysta z typu, który jest zdefiniowany w CS1705b.cs (w wersji 2.0). Powoduje błąd kompilatora CS1705, ponieważ typ ma numer wersji dla CS1705.dll, która jest wyższa niż wersja przywoływanego CS1705.dll.  
  
```csharp  
// CS1705d.cs  
  
// Compile by using the following command:  
//     csc /reference:C:\\CS1705.dll /reference:CS1705c.dll CS1705d.cs  
  
// C:\\CS1705.dll is version 1.0 of the assembly.  
  
class Tester   
{  
    static void Main()  
    {  
        // Return1A returns a type defined in version 1.0.  
        ClassC.Return1A().Method1();  
        // Return1B returns a type defined in version 2.0.  
        ClassC.Return1B().Method1();  
    }  
}  
```  
  
 Można rozwiązać problem w jednym z następujących sposobów:  
  
-   Należy zaktualizować kod, tak aby wszystkie pliki, używają tej samej wersji biblioteki DLL.  
  
-   Dodaj odwołanie do biblioteki DLL w wersji 2.0 do CS1705d.cs przy użyciu następującego polecenia do skompilowania:  
  
     `csc /reference:C:\\CS1705.dll /reference:CS1705.dll /reference:CS1705c.dll CS1705d.cs`  
  
     Mimo że program kompiluje, korzystając z tego polecenia, nadal nie działa. Aby włączyć program do uruchomienia, możesz podać plik konfiguracji aplikacji, która obejmuje [ \<dependentAssembly > element](../../../framework/configure-apps/file-schema/runtime/dependentassembly-element.md) , który używa [ \<assemblyIdentity >](../../../framework/configure-apps/file-schema/runtime/assemblyidentity-element-for-runtime.md) i [ \<codeBase >](../../../framework/configure-apps/file-schema/runtime/codebase-element.md) elementów podrzędnych, aby określić lokalizację biblioteki DLL w wersji 1.0. Aby uzyskać więcej informacji na temat plików konfiguracyjnych, zobacz [konfigurowania aplikacji](../../../framework/configure-apps/index.md).
