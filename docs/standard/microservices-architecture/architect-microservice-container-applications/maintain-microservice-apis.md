# <a name="creating-evolving-and-versioning-microservice-apis-and-contracts"></a><span data-ttu-id="46123-103">Tworzenie, rozwijającymi i przechowywanie wersji mikrousługi interfejsów API i kontrakty</span><span class="sxs-lookup"><span data-stu-id="46123-103">Creating, evolving, and versioning microservice APIs and contracts</span></span>

<span data-ttu-id="46123-104">Mikrousługi interfejsu API jest Umowa między usługą i klientów.</span><span class="sxs-lookup"><span data-stu-id="46123-104">A microservice API is a contract between the service and its clients.</span></span> <span data-ttu-id="46123-105">Można sformułować mikrousługi niezależnie tylko wtedy, gdy nie dzielone jej kontrakt interfejsu API, dlatego tak ważne jest kontraktu.</span><span class="sxs-lookup"><span data-stu-id="46123-105">You will be able to evolve a microservice independently only if you do not break its API contract, which is why the contract is so important.</span></span> <span data-ttu-id="46123-106">W przypadku zmiany umowy, będzie miało wpływ Twoje aplikacje klienckie lub bramy interfejsu API.</span><span class="sxs-lookup"><span data-stu-id="46123-106">If you change the contract, it will impact your client applications or your API Gateway.</span></span>

<span data-ttu-id="46123-107">Rodzaj definicji interfejsu API zależy od tego, na który protokół używasz.</span><span class="sxs-lookup"><span data-stu-id="46123-107">The nature of the API definition depends on which protocol you are using.</span></span> <span data-ttu-id="46123-108">Na przykład jeśli używasz obsługi komunikatów (takich jak [AMQP](https://www.amqp.org/)), interfejsu API składa się z typów komunikatów.</span><span class="sxs-lookup"><span data-stu-id="46123-108">For instance, if you are using messaging (like [AMQP](https://www.amqp.org/)), the API consists of the message types.</span></span> <span data-ttu-id="46123-109">Jeśli korzystasz z usług RESTful i HTTP, interfejs API składa się z adresów URL i formatów żądania i odpowiedzi w formacie JSON.</span><span class="sxs-lookup"><span data-stu-id="46123-109">If you are using HTTP and RESTful services, the API consists of the URLs and the request and response JSON formats.</span></span>

<span data-ttu-id="46123-110">Jednak nawet jeśli jesteś przemyślane o początkowej umowy, interfejs API usługi należy zmieniać wraz z upływem czasu.</span><span class="sxs-lookup"><span data-stu-id="46123-110">However, even if you are thoughtful about your initial contract, a service API will need to change over time.</span></span> <span data-ttu-id="46123-111">Jeśli tak się stanie — i szczególnie w przypadku interfejsu API publiczny interfejs API, używane przez aplikacje klienckie wielu — zwykle nie może wymusić wszystkich klientów do uaktualnienia do kontraktu do interfejsu API.</span><span class="sxs-lookup"><span data-stu-id="46123-111">When that happens—and especially if your API is a public API consumed by multiple client applications—you typically cannot force all clients to upgrade to your new API contract.</span></span> <span data-ttu-id="46123-112">Zazwyczaj konieczne przyrostowo wdrażanie nowych wersji usługi w taki sposób, aby starych i nowych wersji kontraktu usługi są uruchomione jednocześnie.</span><span class="sxs-lookup"><span data-stu-id="46123-112">You usually need to incrementally deploy new versions of a service in a way that both old and new versions of a service contract are running simultaneously.</span></span> <span data-ttu-id="46123-113">W związku z tym należy mieć strategii dla Twojej wersji usługi.</span><span class="sxs-lookup"><span data-stu-id="46123-113">Therefore, it is important to have a strategy for your service versioning.</span></span>

<span data-ttu-id="46123-114">Podczas zmiany interfejsu API są małe, takich jak Jeśli musisz dodać atrybuty lub parametry do interfejsu API, klientów, którzy używają interfejsu API starszych należy przełącznika i korzystać z nowych wersji usługi.</span><span class="sxs-lookup"><span data-stu-id="46123-114">When the API changes are small, like if you add attributes or parameters to your API, clients that use an older API should switch and work with the new version of the service.</span></span> <span data-ttu-id="46123-115">Można podać wartości domyślnej dla brakujących atrybutów, które są wymagane, a klienci można zignorować wszelkie atrybuty dodatkowe odpowiedzi.</span><span class="sxs-lookup"><span data-stu-id="46123-115">You might be able to provide default values for any missing attributes that are required, and the clients might be able to ignore any extra response attributes.</span></span>

<span data-ttu-id="46123-116">Jednak czasami trzeba wprowadzić zmiany głównych i niezgodna z interfejsem API usługi.</span><span class="sxs-lookup"><span data-stu-id="46123-116">However, sometimes you need to make major and incompatible changes to a service API.</span></span> <span data-ttu-id="46123-117">Ponieważ nie można wymusić klienta aplikacji lub usług, aby natychmiast uaktualnić do nowej wersji, usługa musi obsługiwać starszych wersji interfejsu API dla pewnym okresie.</span><span class="sxs-lookup"><span data-stu-id="46123-117">Because you might not be able to force client applications or services to upgrade immediately to the new version, a service must support older versions of the API for some period.</span></span> <span data-ttu-id="46123-118">Jeśli używane są oparte na protokole HTTP mechanizmu, takiego jak REST, jednym z podejść jest osadzić numer wersji interfejsu API w adresie URL lub w nagłówku HTTP.</span><span class="sxs-lookup"><span data-stu-id="46123-118">If you are using an HTTP-based mechanism such as REST, one approach is to embed the API version number in the URL or into a HTTP header.</span></span> <span data-ttu-id="46123-119">Następnie można zdecydować, między implementowania obie wersje usługi jednocześnie w ramach tego samego wystąpienia usługi lub wdrażanie różnych wystąpień, czy obsługują wersja interfejsu API.</span><span class="sxs-lookup"><span data-stu-id="46123-119">Then you can decide between implementing both versions of the service simultaneously within the same service instance, or deploying different instances that each handle a version of the API.</span></span> <span data-ttu-id="46123-120">Dobrym rozwiązaniem dla tego jest [wzorzec mediatora](https://en.wikipedia.org/wiki/Mediator_pattern) (na przykład [biblioteki MediatR](https://github.com/jbogard/MediatR)) rozdzielenie do obsługi niezależnie od wersji inną implementację.</span><span class="sxs-lookup"><span data-stu-id="46123-120">A good approach for this is the [Mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) (for example, [MediatR library](https://github.com/jbogard/MediatR)) to decouple the different implementation versions into independent handlers.</span></span>

<span data-ttu-id="46123-121">Na koniec, w przypadku korzystania z architektury REST [hipermedialnych](https://www.infoq.com/articles/mark-baker-hypermedia) jest najlepszym rozwiązaniem dla wersji usług, dzięki czemu evolvable interfejsów API.</span><span class="sxs-lookup"><span data-stu-id="46123-121">Finally, if you are using a REST architecture, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) is the best solution for versioning your services and allowing evolvable APIs.</span></span>
