# <a name="logical-architecture-versus-physical-architecture"></a><span data-ttu-id="bcb1f-103">Architektura logiczna a architektura fizyczna</span><span class="sxs-lookup"><span data-stu-id="bcb1f-103">Logical architecture versus physical architecture</span></span>

<span data-ttu-id="bcb1f-104">Jest to przydatne w tym momencie zatrzymać, a omówić rozróżnienia między Architektura logiczna a architektura fizyczna i jak to ma zastosowanie do projektowania aplikacji opartych na mikrousługach.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-104">It is useful at this point to stop and discuss the distinction between logical architecture and physical architecture, and how this applies to the design of microservice-based applications.</span></span>

<span data-ttu-id="bcb1f-105">Aby rozpocząć, tworzenia mikrousług nie wymaga użycia każdej określonej technologii.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-105">To begin, building microservices does not require the use of any specific technology.</span></span> <span data-ttu-id="bcb1f-106">Na przykład kontenery platformy Docker nie są wymagane, aby można było utworzyć opartych na mikrousługach architektury.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-106">For instance, Docker containers are not mandatory in order to create a microservice-based architecture.</span></span> <span data-ttu-id="bcb1f-107">Te mikrousług może także uruchomić jako zwykły procesy.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-107">Those microservices could also be run as plain processes.</span></span> <span data-ttu-id="bcb1f-108">Mikrousługi to architektura logiczna.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-108">Microservices is a logical architecture.</span></span>

<span data-ttu-id="bcb1f-109">Ponadto, nawet wtedy, gdy mikrousługi można fizycznie zaimplementować jako pojedynczą usługę, procesu lub kontenera (sake firmy prostotę, to podejście w pierwotnej wersji [ramach aplikacji eShopOnContainers](https://aka.ms/MicroservicesArchitecture)), to parzystość mikrousługi biznesowych i fizycznych usługi lub kontenera nie jest zawsze wymagany we wszystkich przypadkach podczas kompilowania dużych i złożonych aplikacji, składające się z wielu dziesiątek, jak i nawet setki usług.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-109">Moreover, even when a microservice could be physically implemented as a single service, process, or container (for simplicity’s sake, that is the approach taken in the initial version of [eShopOnContainers](https://aka.ms/MicroservicesArchitecture)), this parity between business microservice and physical service or container is not necessarily required in all cases when you build a large and complex application composed of many dozens or even hundreds of services.</span></span>

<span data-ttu-id="bcb1f-110">To jest, gdy istnieje różnica między Architektura logiczna a architektura fizyczna aplikacji.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-110">This is where there is a difference between an application’s logical architecture and physical architecture.</span></span> <span data-ttu-id="bcb1f-111">Architektura logiczna a logiczne granice systemu nie są zawsze mapowane jeden do jednego z architekturą fizycznej lub wdrożenia.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-111">The logical architecture and logical boundaries of a system do not necessarily map one-to-one to the physical or deployment architecture.</span></span> <span data-ttu-id="bcb1f-112">Może się zdarzyć, ale często nie.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-112">It can happen, but it often does not.</span></span>

<span data-ttu-id="bcb1f-113">Chociaż może być zidentyfikowano niektórych mikrousług firmy lub ograniczonych kontekstów, nie oznacza to, zawsze jest najlepszym sposobem ich wdrażania, tworząc pojedynczą usługę (np. interfejsu API sieci Web platformy ASP.NET) lub pojedynczym kontenerem platformy Docker dla poszczególnych mikrousług biznesowych.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-113">Although you might have identified certain business microservices or Bounded Contexts, it does not mean that the best way to implement them is always by creating a single service (such as an ASP.NET Web API) or single Docker container for each business microservice.</span></span> <span data-ttu-id="bcb1f-114">Reguła informujący o tym, każda mikrousługa firm ma być implementowane przy użyciu jednej usługi lub kontener jest zbyt sztywne.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-114">Having a rule saying each business microservice has to be implemented using a single service or container is too rigid.</span></span>

<span data-ttu-id="bcb1f-115">Dlatego mikrousług biznesowych lub ograniczony kontekst jest logiczną architekturę, która może być pokrywają się (lub nie) przy użyciu architektury fizycznych.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-115">Therefore, a business microservice or Bounded Context is a logical architecture that might coincide (or not) with physical architecture.</span></span> <span data-ttu-id="bcb1f-116">Istotną kwestią jest to, że mikrousług biznesowych lub ograniczony kontekst musi być autonomiczne, umożliwiając kodu i stanie się oddzielnie wersjonowanych, wdrażanie i skalowanie.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-116">The important point is that a business microservice or Bounded Context must be autonomous by allowing code and state to be independently versioned, deployed, and scaled.</span></span>

<span data-ttu-id="bcb1f-117">Jak pokazano na rysunku 4 – 8, mikrousługi firm katalogu może składać się z kilku usług lub procesów.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-117">As Figure 4-8 shows, the catalog business microservice could be composed of several services or processes.</span></span> <span data-ttu-id="bcb1f-118">Mogą to być wiele usług interfejsu API sieci Web platformy ASP.NET lub dowolny inny rodzaj usługi przy użyciu protokołu HTTP lub innych protokołów.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-118">These could be multiple ASP.NET Web API services or any other kind of services using HTTP or any other protocol.</span></span> <span data-ttu-id="bcb1f-119">Co ważniejsze usług można udostępnić te same dane, tak długo, jak te usługi są spójne w odniesieniu do tej samej domeny biznesowej.</span><span class="sxs-lookup"><span data-stu-id="bcb1f-119">More importantly, the services could share the same data, as long as these services are cohesive with respect to the same business domain.</span></span>
