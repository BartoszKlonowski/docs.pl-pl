# <a name="microservices-architecture"></a><span data-ttu-id="a7047-103">Architektura Mikrousług</span><span class="sxs-lookup"><span data-stu-id="a7047-103">Microservices architecture</span></span>

<span data-ttu-id="a7047-104">Jak wskazuje nazwa, architektura mikrousług jest podejście do kompilowania aplikacji serwera jako zestaw małych usług.</span><span class="sxs-lookup"><span data-stu-id="a7047-104">As the name implies, a microservices architecture is an approach to building a server application as a set of small services.</span></span> <span data-ttu-id="a7047-105">Oznacza to, że architektura mikrousług jest zorientowany głównie na zaplecze, chociaż to podejście jest również używany fronton.</span><span class="sxs-lookup"><span data-stu-id="a7047-105">That means a microservices architecture is mainly oriented to the back-end, although the approach is also being used for the front end.</span></span> <span data-ttu-id="a7047-106">Każda usługa jest uruchamiana we własnym procesie i komunikuje się z innymi procesami przy użyciu protokołów, takich jak HTTP/HTTPS, funkcja WebSockets, lub [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span><span class="sxs-lookup"><span data-stu-id="a7047-106">Each service runs in its own process and communicates with other processes using protocols such as HTTP/HTTPS, WebSockets, or [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span></span> <span data-ttu-id="a7047-107">Każda mikrousługa implementuje określonej domeny end-to-end lub funkcję biznesową w obrębie granicy kontekstu, a każda musi być opracowana autonomicznie i być niezależnie do wdrożenia.</span><span class="sxs-lookup"><span data-stu-id="a7047-107">Each microservice implements a specific end-to-end domain or business capability within a certain context boundary, and each must be developed autonomously and be deployable independently.</span></span> <span data-ttu-id="a7047-108">Na koniec poszczególne mikrousługi powinny odpowiadać jej modelu danych pokrewne domeny i logika domeny (niezależność danych i zarządzania danymi zdecentralizowane) i mogą być oparte na różne technologie przechowywania danych (SQL, NoSQL) i w różnych językach programowania.</span><span class="sxs-lookup"><span data-stu-id="a7047-108">Finally, each microservice should own its related domain data model and domain logic (sovereignty and decentralized data management) and could be based on different data storage technologies (SQL, NoSQL) and different programming languages.</span></span>

<span data-ttu-id="a7047-109">Rozmiar powinien być mikrousługi?</span><span class="sxs-lookup"><span data-stu-id="a7047-109">What size should a microservice be?</span></span> <span data-ttu-id="a7047-110">Podczas tworzenia mikrousług, rozmiar nie powinien być istotne.</span><span class="sxs-lookup"><span data-stu-id="a7047-110">When developing a microservice, size shouldn't be the important point.</span></span> <span data-ttu-id="a7047-111">Zamiast tego należy koniecznie zwrócić należy utworzyć luźno sprzężonych usług, więc trzeba autonomii programowania, wdrażania i skalowania dla każdej usługi.</span><span class="sxs-lookup"><span data-stu-id="a7047-111">Instead, the important point should be to create loosely coupled services so you have autonomy of development, deployment, and scale, for each service.</span></span> <span data-ttu-id="a7047-112">Oczywiście i projektowanie mikrousług, należy spróbować były tak małej, jak to możliwe tak długo, jak nie ma zbyt wiele zależności bezpośrednich za pomocą innych mikrousług.</span><span class="sxs-lookup"><span data-stu-id="a7047-112">Of course, when identifying and designing microservices, you should try to make them as small as possible as long as you don't have too many direct dependencies with other microservices.</span></span> <span data-ttu-id="a7047-113">Ważniejsze niż rozmiar mikrousług jest to wewnętrzny spójności, który musi mieć oraz jego niezależność od innych usług.</span><span class="sxs-lookup"><span data-stu-id="a7047-113">More important than the size of the microservice is the internal cohesion it must have and its independence from other services.</span></span>

<span data-ttu-id="a7047-114">Dlaczego architektura mikrousług?</span><span class="sxs-lookup"><span data-stu-id="a7047-114">Why a microservices architecture?</span></span> <span data-ttu-id="a7047-115">Krótko mówiąc zapewnia elastyczność długoterminowego.</span><span class="sxs-lookup"><span data-stu-id="a7047-115">In short, it provides long-term agility.</span></span> <span data-ttu-id="a7047-116">Mikrousługi Włącz lepsze łatwość konserwacji w złożonych, duże i wysoce skalowalnych systemów, umożliwiając tworzenie aplikacji na podstawie wielu usług niezależnie do wdrożenia, w których każdy może mieć cykle życia szczegółową i autonomicznych.</span><span class="sxs-lookup"><span data-stu-id="a7047-116">Microservices enable better maintainability in complex, large, and highly-scalable systems by letting you create applications based on many independently deployable services that each have granular and autonomous lifecycles.</span></span>

<span data-ttu-id="a7047-117">Jako dodatkowa korzyść mikrousługi można skalować niezależnie.</span><span class="sxs-lookup"><span data-stu-id="a7047-117">As an additional benefit, microservices can scale out independently.</span></span> <span data-ttu-id="a7047-118">Zamiast pojedynczej aplikacji monolitycznych, należy skalować jako jednostkę, można zamiast tego skalować określonych mikrousług.</span><span class="sxs-lookup"><span data-stu-id="a7047-118">Instead of having a single monolithic application that you must scale out as a unit, you can instead scale out specific microservices.</span></span> <span data-ttu-id="a7047-119">W ten sposób można skalować tylko obszar funkcjonalny, który wymaga więcej przetwarzania zasilania lub sieci przepustowość do obsługi żądanie, zamiast skalowania w poziomie innych obszarów aplikacji, które nie muszą być skalowane.</span><span class="sxs-lookup"><span data-stu-id="a7047-119">That way, you can scale just the functional area that needs more processing power or network bandwidth to support demand, rather than scaling out other areas of the application that don't need to be scaled.</span></span> <span data-ttu-id="a7047-120">Oznacza to oszczędności, ponieważ potrzebne mniejszą sprzętu.</span><span class="sxs-lookup"><span data-stu-id="a7047-120">That means cost savings because you need less hardware.</span></span>

<span data-ttu-id="a7047-126">Tworzenie szczegółowych aplikacji mikrousług umożliwia ciągłą integrację i ciągłe dostarczanie rozwiązań.</span><span class="sxs-lookup"><span data-stu-id="a7047-126">Architecting fine-grained microservices-based applications enables continuous integration and continuous delivery practices.</span></span> <span data-ttu-id="a7047-127">Przyspiesza również dostarczania nowych funkcji do aplikacji.</span><span class="sxs-lookup"><span data-stu-id="a7047-127">It also accelerates delivery of new functions into the application.</span></span> <span data-ttu-id="a7047-128">Szczegółowych skład aplikacji umożliwia uruchamianie i testowanie mikrousług w izolacji i autonomicznie ewolucja przy zachowaniu wyczyść umów między nimi.</span><span class="sxs-lookup"><span data-stu-id="a7047-128">Fine-grained composition of applications also allows you to run and test microservices in isolation, and to evolve them autonomously while maintaining clear contracts between them.</span></span> <span data-ttu-id="a7047-129">Tak długo, jak nie zmieniaj interfejsy lub umowy, można zmienić wewnętrzną implementację wszystkie mikrousługi, lub Dodaj nowe funkcje bez przerywania innych mikrousług.</span><span class="sxs-lookup"><span data-stu-id="a7047-129">As long as you don't change the interfaces or contracts, you can change the internal implementation of any microservice or add new functionality without breaking other microservices.</span></span>

<span data-ttu-id="a7047-130">Poniżej przedstawiono ważne kwestie umożliwiające sukces w przejściem do środowiska produkcyjnego w systemie opartych na mikrousługach:</span><span class="sxs-lookup"><span data-stu-id="a7047-130">The following are important aspects to enable success in going into production with a microservices-based system:</span></span>

- <span data-ttu-id="a7047-131">Monitorowanie i kontrole kondycji usług i infrastruktury.</span><span class="sxs-lookup"><span data-stu-id="a7047-131">Monitoring and health checks of the services and infrastructure.</span></span>

- <span data-ttu-id="a7047-132">Skalowalną infrastrukturę usługi (czyli chmury i koordynatorów).</span><span class="sxs-lookup"><span data-stu-id="a7047-132">Scalable infrastructure for the services (that is, cloud and orchestrators).</span></span>

- <span data-ttu-id="a7047-133">Bezpieczeństwo projektowania i implementacji, na różnych poziomach: uwierzytelnianie, autoryzacja, zarządzania wpisami tajnymi, bezpiecznej komunikacji, itp.</span><span class="sxs-lookup"><span data-stu-id="a7047-133">Security design and implementation at multiple levels: authentication, authorization, secrets management, secure communication, etc.</span></span>

- <span data-ttu-id="a7047-134">Dostarczanie aplikacji szybkie, zwykle przy użyciu różnych zespołów, skupiając się na różnych mikrousług.</span><span class="sxs-lookup"><span data-stu-id="a7047-134">Rapid application delivery, usually with different teams focusing on different microservices.</span></span>

- <span data-ttu-id="a7047-135">Praktyki DevOps i ciągłej integracji/ciągłego Dostarczania i infrastruktury.</span><span class="sxs-lookup"><span data-stu-id="a7047-135">DevOps and CI/CD practices and infrastructure.</span></span>
