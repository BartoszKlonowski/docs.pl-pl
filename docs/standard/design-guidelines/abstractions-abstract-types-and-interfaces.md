# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="93402-102">Abstrakcje (typy abstrakcyjne i interfejsy)</span><span class="sxs-lookup"><span data-stu-id="93402-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="93402-103">Streszczenie to typ, który opisuje kontrakt, ale nie zapewnia pełnej implementacji kontraktu.</span><span class="sxs-lookup"><span data-stu-id="93402-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="93402-104">Streszczenia są zwykle implementowane jako klasy abstrakcyjne lub interfejsy i są one ze dobrze zdefiniowanym zestawem dokumentacji referencyjnej opisującym wymaganą semantykę typów implementujących kontrakt.</span><span class="sxs-lookup"><span data-stu-id="93402-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="93402-105">Niektóre z najważniejszych streszczeń w .NET Framework obejmują <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>i <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="93402-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>

 <span data-ttu-id="93402-106">Struktury można rozciągnąć przez implementację konkretnego typu, który obsługuje umowę abstrakcji i używa tego konkretnego typu z interfejsami API platformy, które używają (działającego na) abstrakcji.</span><span class="sxs-lookup"><span data-stu-id="93402-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>

 <span data-ttu-id="93402-107">Zrozumiałe i przydatne streszczenie, które jest w stanie przetrzymywać testy czasu, są bardzo trudne do zaprojektowania.</span><span class="sxs-lookup"><span data-stu-id="93402-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="93402-108">Głównym trudnością jest uzyskanie właściwego zestawu elementów członkowskich, nie więcej i mniej.</span><span class="sxs-lookup"><span data-stu-id="93402-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="93402-109">Jeśli streszczenie ma zbyt wiele elementów członkowskich, stanie się trudne lub nawet niemożliwe do zaimplementowania.</span><span class="sxs-lookup"><span data-stu-id="93402-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="93402-110">Jeśli ma za mało elementów członkowskich dla pozostałej funkcjonalności, będzie bezużyteczny w wielu interesujących scenariuszach.</span><span class="sxs-lookup"><span data-stu-id="93402-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>

 <span data-ttu-id="93402-111">Zbyt wiele abstrakcji w strukturze również negatywnie wpływa na użyteczność struktury.</span><span class="sxs-lookup"><span data-stu-id="93402-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="93402-112">Często trudno jest zrozumieć abstrakcję bez zrozumienia, jak pasuje do większego obrazu konkretnych implementacji i interfejsów API działających na potrzeby abstrakcji.</span><span class="sxs-lookup"><span data-stu-id="93402-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="93402-113">Ponadto nazwy abstrakcji i ich składowe są niekoniecznie abstrakcyjne, co często sprawia, że są one tajemnicze i nieodpowiednie, bez uprzedniego poznania szerszego kontekstu ich użycia.</span><span class="sxs-lookup"><span data-stu-id="93402-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>

 <span data-ttu-id="93402-114">Jednakże abstrakcje zapewniają wyjątkowo zaawansowane rozszerzalność, która nie jest często zgodna z innymi mechanizmami rozszerzalności.</span><span class="sxs-lookup"><span data-stu-id="93402-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="93402-115">Są one podstawą wielu wzorców architektonicznych, takich jak wtyczki, niewersja kontroli (IoC), potoki itd.</span><span class="sxs-lookup"><span data-stu-id="93402-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="93402-116">Są one również niezwykle ważne do testowania struktur.</span><span class="sxs-lookup"><span data-stu-id="93402-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="93402-117">Dobre abstrakcje sprawiają, że istnieje możliwość wypróbowania dużych zależności na potrzeby testowania jednostkowego.</span><span class="sxs-lookup"><span data-stu-id="93402-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="93402-118">Podsumowując, abstrakcje są odpowiedzialne za pomyślne zamiar — po rozbudowaniu nowoczesnych platform zorientowanych obiektowo.</span><span class="sxs-lookup"><span data-stu-id="93402-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>

 <span data-ttu-id="93402-119">❌ nie zapewniają abstrakcji, chyba że są testowane przez opracowywanie kilku konkretnych implementacji i interfejsów API korzystających z abstrakcji.</span><span class="sxs-lookup"><span data-stu-id="93402-119">❌ DO NOT provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>

 <span data-ttu-id="93402-120">✔️ NALEŻY uważnie wybierać między klasą abstrakcyjną i interfejsem podczas projektowania abstrakcji.</span><span class="sxs-lookup"><span data-stu-id="93402-120">✔️ DO choose carefully between an abstract class and an interface when designing an abstraction.</span></span>

 <span data-ttu-id="93402-121">✔️ ROZWAŻYĆ dostarczenie testów referencyjnych dla konkretnych implementacji abstrakcji.</span><span class="sxs-lookup"><span data-stu-id="93402-121">✔️ CONSIDER providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="93402-122">Takie testy powinny pozwolić użytkownikom na przetestowanie, czy ich implementacje prawidłowo implementują kontrakt.</span><span class="sxs-lookup"><span data-stu-id="93402-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
