## <a name="unmanaged-code-interoperability"></a><span data-ttu-id="acec5-128">Współdziałanie z kodem niezarządzanym</span><span class="sxs-lookup"><span data-stu-id="acec5-128">Unmanaged code interoperability</span></span>

<span data-ttu-id="acec5-129">Oczywiście CLR umożliwia przekazywanie granice między world zarządzane i niezarządzane, a wiele kod, który obsługuje, nawet w [podstawowej biblioteki klas](framework-libraries.md).</span><span class="sxs-lookup"><span data-stu-id="acec5-129">Of course, the CLR allows passing the boundaries between managed and unmanaged world, and there is a lot of code that does that, even in the [Base Class Libraries](framework-libraries.md).</span></span> <span data-ttu-id="acec5-130">Ta metoda jest wywoływana **współdziałanie** lub po prostu **interop** skrócie.</span><span class="sxs-lookup"><span data-stu-id="acec5-130">This is called **interoperability** or just **interop** for short.</span></span> <span data-ttu-id="acec5-131">Umożliwia te przepisy, na przykład dobiega końca niezarządzane biblioteki i wywołują go.</span><span class="sxs-lookup"><span data-stu-id="acec5-131">These provisions would allow you to, for example, wrap up an unmanaged library and call into it.</span></span> <span data-ttu-id="acec5-132">Jest jednak należy pamiętać, że po tym, gdy kod przekazuje granice środowiska uruchomieniowego rzeczywiste Zarządzanie wykonywania jest ponownie ręczne kodu niezarządzanego, a w związku z tym podlega takie same ograniczenia.</span><span class="sxs-lookup"><span data-stu-id="acec5-132">However, it is important to note that once you do this, when the code passes the boundaries of the runtime, the actual management of the execution is again in the hand of unmanaged code, and thus falls under the same restrictions.</span></span>
